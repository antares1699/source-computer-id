Dugaan korupsi file palsu pada Windows 7

Baru-baru ini Laptop saya terkadang memperingatkan tentang file yang rusak pada hard drive (Samsung SSD PB22-JS3 TM). Ini hanya terjadi sejauh ini ketika memperbarui (atau memeriksa) repositori SVN dengan TortoiseSVN atau baris perintah klien Subversion.

Yang menyenangkan adalah bahwa file yang rusak selalu menjadi direktori .svn (meskipun entri direktori mungkin berisi file dalam direktori itu juga, jika ukurannya cukup kecil - yang seharusnya terjadi pada SVN). Namun, ketika melihat ke dalam direktori yang diperingatkan saya melihat tidak ada yang aneh atau tidak biasa dan tidak mendapatkan peringatan lagi tentangnya dan mencoba lagi (SVN berhenti memperbarui begitu kesalahan itu terjadi - TortoiseSVN bahkan dengan pesan kesalahan yang sesuai) memperbarui pekerjaan copy berfungsi (baik, sebagian besar; kadang-kadang melakukannya lagi, meskipun dengan direktori yang berbeda).

Karena laptop baru berumur beberapa bulan, saya ragu SSD-nya sudah rusak — penggunaan normal selama lima bulan seharusnya tidak terlalu mengejutkan. Juga (sejauh ini) hanya terjadi dengan pembaruan SVN pada repositori besar. Mungkin itu terlalu banyak penulisan dalam waktu singkat dan beberapa bagian antara perangkat lunak dan perangkat keras tidak cukup cepat menyusul - saya tidak cukup tahu tentang ini untuk benar-benar membuat tebakan yang tepat di sini.

Ada yang tahu ada apa di sini?

ETA: Catatan untuk ditambahkan: Saya telah menjalankan chkdsk (tampaknya tetap menjadwalkan sendiri ketika ini terjadi) dan tidak menemukan sesuatu yang luar biasa.

Lihat jika ini terkait dengan masalah Anda.

Ringkasan singkat: Driver sistem file Windows 7 NTFS memiliki bug yang terkadang ketika suatu proses mencoba mengakses file yang sedang digunakan oleh proses lain (seperti layanan pengindeksan file latar belakang) tidak menghasilkan kesalahan "Access Denied" yang akan menjadi kasus normal melainkan "File Rusak". Hal ini pada gilirannya juga menyebabkan chkdsk dijadwalkan untuk startup sistem berikutnya. Mengutip dari artikel tertaut:

Microsoft telah dikonfirmasi bug ini:

Ini adalah regresi yang diketahui di Windows 7 di sistem file NTFS. Ini terjadi saat melakukan penggantian nama pada file yang memiliki oplock atom di atasnya (oplock atom adalah fitur baru di Windows 7). Pengindeks menggunakan atomic oplock, itulah mengapa membantu saat Anda menonaktifkan pengindeks. Explorer juga menggunakan atomic oplock, itulah sebabnya Anda masih melihat masalah ini. Ketika ini terjadi STATUS_FILE_CORRUPT dikembalikan secara tidak benar dan volume ditandai "kotor" yang merupakan sinyal ke sistem bahwa chkdsk perlu dijalankan. Sebenarnya tidak ada korupsi yang terjadi.

Saya mengalami masalah serupa dengan Samsung Drive yang sama (MMCRE28G8MXP-0VBL1 - Saya cukup yakin itu dari seri PB22)

Saya hanya memilikinya selama dua minggu, tetapi startup normal sudah terganggu untuk menjalankan chkdisk untuk konsistensi drive beberapa kali. Tidak ada masalah yang muncul, tetapi ini pasti menanggapi masalah korupsi NTFS. Di Peraga Peristiwa, yang menunjukkan kepada saya beberapa galat korupsi NTFS - ID Peristiwa: 55 Kategori tugas 2.

Saya tidak bisa mengatakan dengan pasti, tetapi saya merasa bahwa kesalahan tersebut bertepatan dengan penggunaan drive dengan aktivitas tinggi. Saya telah bekerja dengan beberapa file Excel besar (10MB +) secara bersamaan dan jelas bahwa drive tersebut bekerja cukup keras untuk mengimbanginya. Mungkin NTFS menjadi "tidak sinkron"

Ketika dijalankan melalui chkdisk, ia menyebutkan Jurnal USN, tapi saya belum memperhatikannya.
