Beberapa contoh Sublime menggunakan kredensial windows yang berbeda?

Sebagai seorang profesional yang berpikiran keamanan, saya melakukan pekerjaan sehari-hari di komputer saya sebagai akun yang tidak memiliki hak istimewa. Saya akan memulai program individual, shell, atau sebagainya dengan akun Administrator Domain saya yang memiliki hak istimewa.

Saya hampir selalu membuka editor luhur saya di akun saya yang tidak memiliki hak istimewa dengan selusin file terbuka. Kadang-kadang saya ingin membuka Jendela luhur baru dengan hak istimewa yang ditinggikan (Mulai -> Klik kanan pada Sublime -> Jalankan sebagai pengguna berbeda). Sayangnya ini tidak memulai instance luhur baru, ini hanya mem-flash instance tanpa hak istimewa yang ada.

Ada lusinan hits di Google tentang bagaimana Anda dapat menggunakan File-> Jendela Baru atau pintasan untuk membuka contoh baru, tetapi ini tidak membantu saya sama sekali. Karena metode itu tidak akan membuat jendela berjalan di bawah serangkaian hak istimewa yang berbeda.

Jadi, bagaimana cara memulai contoh kedua Sublime berjalan dengan hak istimewa dari akun lain?

Sistem saya saat ini menjalankan Windows 8.1, dan saya menggunakan build 3083 sublim, tetapi saya rasa ini tidak akan terlalu menjadi masalah.

Secara teknis, cara untuk mencapai efek tersebut adalah dengan membuka objek sinkronisasi global yang dapat diakses seperti pipa bernama. Proses kedua akan mendeteksi objek yang ada dan dengan cara tertentu memberi tahu proses pertama untuk membuka file.

Dan itulah tepatnya yang dilakukan Sublime Text. Anda bisa mengamati ini dengan Proses Explorer (SysInternals) :

  1. Jalankan Process Explorer sebagai administrator
  2. Jalankan Sublime text
  3. Pilih Sublime text dalam daftar proses
  4. tekan Ctrl + H untuk memperlihatkan panel bawah untuk tuas
  5. Temukan pegangan tipe Mutan dengan "Sublime Text 2" di namanya. Ini mungkin terlihat seperti \Sessions\1\BaseNamedObjects\4d3560c7bb75b0aede072672a3c001bb-Sublime Text 2
  6. Klik kanan Mutan
  7. Pilih "Tutup Handle"
  8. Mulai contoh lain dari Sublime Text

Sekarang Anda tahu cara memulai contoh baru Sublime Text. Tentu Anda ingin mengotomatiskan proses ini. Alur urutan yang diperlukan sekarang diketahui oleh Anda:

  1. Temukan semua proses sublim_text.exe, mis. menggunakan Proses Enum (MSDN)
  2. Periksa daftar pegangan mereka, mis. menggunakan NtQuerySystemInformation dengan SystemHandleInformation
  3. Tutup semua Mutan dengan nama "Sublime Test 2", mis. menggunakan CloseHandle (MSDN) .

Anda juga harus menutup pegangan untuk proses baru, karena proses baru akan membuatnya lagi.

Tangani (Sysinternals) sangat membantu untuk menyiapkan skrip yang mencapai apa yang Anda butuhkan. Itu dapat menemukan pegangan (baris 1) dan menutup pegangan (baris 2)

Handle -p sublime_text.exe -a Sublime | find "Mutant"
Handle -c <handle> -p <pid> -y

Dalam skrip lengkap berikut, sesuaikan nama pengguna. Salin handles.exe ke direktori yang sama dan jalankan file batch sebagai administrator (karena handles.exe memerlukan hak admin:

@echo off
REM Just in case this is run multiple times from a command line 
set pid=    
set handle=
REM Make the working directory the directory of the batch file
cd /d %~dp0 
REM Find PID and Handle
for /f "tokens=3,6" %%i in ('handle -p sublime_text.exe -a Sublime -accepteula ^| find "Mutant"') do set pid=%%i & set handle=%%j   
if "%pid%"=="" goto sublime 
REM Close the handle
handle -c %handle:~0,-1% -p %pid% -y > nul  
:sublime
runas /user:Username sublime_text.exe   
if errorlevel 1 pause

Windows tidak menyediakan cara apa pun untuk melakukan apa yang dilakukan Sublime Text secara langsung, yang berarti tidak ada cara universal untuk menimpa perilaku ini. Ini akan tergantung pada bagaimana program itu sendiri mengimplementasikan perilaku ini. Sayangnya, fakta bahwa sesi kedua adalah Admin membuat lebih sulit untuk menggunakan teknik yang memblokir komunikasi di seluruh tingkat hak, karena aplikasi yang lebih istimewa perlu berbicara dengan yang kurang memiliki hak istimewa, dan itu hampir selalu diizinkan.

Pertama-tama, Anda mungkin dapat menonaktifkan penggabungan sesi secara otomatis. Kelemahan dari ini adalah bahwa file yang ingin Anda buka di jendela saat ini akan terbuka di jendela baru. Jadi itu jelas merupakan pilihan yang lebih rendah, tetapi itu ada.

Jika Sublime hanya mencari contoh lain dari dirinya sendiri berdasarkan namanya, Anda dapat membuat dua salinan dari yang dapat dieksekusi, menambahkan "Admin" pada nama salah satunya, dan menggunakan tab Kompatibilitas untuk menandainya sebagai yang selalu perlu dijalankan sebagai Admin. Tambahkan ke Start atau Taskbar Anda, dan Anda memiliki editor khusus admin yang dapat diluncurkan dengan mudah. Perhatikan bahwa itu tidak akan diperbarui ketika yang pertama melakukannya. Menggunakan symlink atau hardlink akan berfungsi jika Sublime memeriksa baris perintah, tetapi tidak jika Sublime memeriksa nama gambar (nama gambar selalu menyelesaikan ke jalur kanonik pertama).

Pendekatan yang sedikit ekstrim yang mungkin berhasil adalah dengan menggunakan desktop jarak jauh loopback. Ini hanya berfungsi pada Server SKU Windows karena memerlukan dua sesi interaktif yang aktif sekaligus (versi klien Windows melarang) tetapi Sublime mungkin berhenti di batas Sesi bahkan di mana biasanya mengabaikan batas Pengguna. Cukup jauh ke localhost dengan kredensial admin domain, lalu luncurkan Sublime (atau bahkan atur sehingga secara otomatis meluncurkan Sublime, atau mungkin bahkan hanya meneruskan satu aplikasi itu kembali ke desktop Anda alih-alih menggambar sepenuhnya desktop lain). Pendekatan ini akan berfungsi untuk hal-hal seperti mutex bernama atau serupa yang dibuat dalam sesi pengguna (bukan secara global).

Ini mungkin bukan file atau kunci registri, karena satu-satunya tempat yang dapat dibuat oleh instance Sublime yang tidak memiliki hak istimewa yang umumnya bukan tempat versi istimewa akan mencarinya. Namun, ada pengecualian; misalnya, ProgramData dapat ditulis oleh dunia. Jika begitulah cara Sublime mendeteksi instance lain dari dirinya sendiri, saya tidak punya saran yang bisa diterapkan kecuali untuk menggunakan editor yang berbeda; itu hanya perilaku datar yang tidak kompatibel dengan banyak pengguna.

Jika pendekatan di atas tidak berhasil, satu-satunya pilihan Anda (singkatnya selalu memulai Sublime sebagai Admin, atau menggunakan editor lain untuk pekerjaan Admin) adalah menyodok pengembang dan meminta mereka untuk berperilaku lebih baik dalam skenario multi-pengguna. Sebagian besar program Windows paling tidak menyadari konsep bahwa banyak pengguna mungkin ingin menjalankan program sekaligus, tetapi beberapa di antaranya sama sekali tidak kompatibel dengannya. Pengembang Sublime dapat, misalnya, memeriksa kredensial yang dijalankan oleh instance yang sudah ada sebelum menggabungkan sesi ... tetapi jika pengembang tidak memikirkannya sendiri, Anda mungkin harus melaporkan bug agar mereka dapat memperbaikinya .

Seperti yang telah disebutkan sebelumnya , hanya memiliki jalur yang berbeda ke Exe of Sublime bekerja di sekitar penggunaan kembali instance yang sedang berjalan dan seseorang bahkan tidak perlu menyalin file, tetapi menautkan Exe menggunakan tautan keras atau direktori instalasi itu sendiri menggunakan Persimpangan jalan bekerja dengan baik.

Selain itu, dalam diskusi serupa di GitHub, argumen baris perintah --multiinstance disebutkan dan itu benar-benar membuat Sublime mengabaikan semua sesi yang sedang berjalan dan memulai yang baru. Jadi yang saya lakukan hanyalah membuat tautan shell yang berisi jalur ke Sublime di satu-satunya direktori instalasi defaultnya, tambahkan argumen baris perintah dan aktifkan untuk memulai sebagai admin. Melihat Monitor Proses, memang ada contoh admin yang berjalan dan terus berjalan. Sejauh ini tidak menemukan kekurangan.
