Bagaimana cara mengetahui jika ada file di sektor buruk tertentu?

Saya mengkloning HDD saya yang gagal dengan Clonezilla dengan opsi --rescue dan mencantumkan banyak bad sector. Beberapa bad sector adalah 3194529280, 3194529792, 3194530304, 3194530816. Bagaimana cara mengetahui file mana yang berada di bad sector tersebut?

Saya mengharapkan jawaban seperti

> whichfileisatsector.exe 3194529280
c:\Users\Chloe\Pictures\cat.jpg

Saya melihat blog yang menyebutkan nfi.exe dengan tautan ke Microsoft, tetapi tidak lagi tersedia dan tautannya mati.

- Anda akan menemukan nfi.exe di sini:
http://www.jumpjet.info/Application-Software/Windows/win2k.htm
Unduh "oem3sr2.zip", ada di sana. Perintahnya adalah:

nfi X: [sector number]
(Replace "X:" with the relevant drive letter.)

Anda juga bisa mendapatkan daftar lengkap semua file di partisi tertentu jika Anda tidak menentukan nomor sektor:

nfi X: >"D:\nfi listing.txt"
(Replace "X:" with the relevant drive letter. Adjust the name/path of the output file.)

(File teks yang dihasilkan bisa sangat besar jika ada banyak file di partisi yang dianalisis. Untuk HDD 3TB yang sedang saya kerjakan, ini mengekstrak file daftar 425MB.)

- Alat fsutil yang sudah ada di Windows (setidaknya Windows 7) juga dapat menyediakan fungsi ini, di antara yang lainnya. Perintahnya adalah:

fsutil volume querycluster X: [cluster number]
(Replace "X:" with the relevant drive letter.)

Di sini input harus dalam cluster, jadi Anda harus membagi nomor sektor dengan jumlah sektor per cluster, biasanya 8 untuk ukuran cluster 4KB umum. Penyajian hasil lebih ramping daripada nfi.exe, yang dapat membingungkan jika file memiliki banyak fragmen (tampaknya satu file dapat disajikan sebagai beberapa nomor file di MFT, dan nfi hanya menampilkan salah satu dari bagian ini, Saya belum begitu mengerti cara kerjanya); juga, ini dapat memberikan nama pendek alih-alih nama lengkap, sehingga membutuhkan langkah tambahan untuk mengidentifikasi file itu dengan pasti (misalnya saya mendapat "2017_0 ~ 3.MP4", dalam folder yang penuh dengan file video dengan nama lengkap dimulai dengan “2017_0”). Dengan fsutil Anda mendapatkan satu baris yang memberi Anda hanya info yang Anda butuhkan, dengan andal:

Cluster 0x000000001ec1ba42 utilisé par ----D \fichiers corrompus\2017_07_2122_49 - Arte - 101 - Depeche Mode.mp4::$DATA

(Nomor cluster selalu ditampilkan dalam format heksadesimal, tetapi Anda dapat mengetiknya dalam format desimal; di sini, saya meminta cluster 516012610. Ada dalam bahasa Prancis karena Saya di Prancis!: ^ P)
Dengan kedua alat tersebut, Anda harus menyediakan file logis sektor atau nomor cluster untuk mendapatkan hasil yang benar, jadi jika Clonezilla memberi Anda daftar mutlak atau fisik nomor sektor (yang mungkin terjadi, meskipun saya belum mencobanya, karena perangkat lunak kloning dirancang untuk menyalin semuanya tanpa memperhatikan skema partisi) Anda harus menghapus offset partisi. Misalnya, dengan hard drive 3TB yang sedang saya kerjakan, hanya ada satu partisi yang dimulai di sektor 264192, jadi jika saya memiliki jumlah absolut dari sektor buruk (disediakan oleh HD Sentinel misalnya - lihat di bawah), katakanlah 4128365072, dan ingin tahu file mana yang dimilikinya, saya harus mengetik 4128365072 - 264192 = 4128100880 dengan nfi.exe, dan 4128100880/8 = 516012610 dengan fsutil.

Alat lain yang saya tahu dapat memberikan informasi semacam itu:

- Hard Disk Sentinel (tidak gratis tetapi ini adalah perangkat lunak terbaik yang saya tahu untuk tujuan penilaian kesehatan hard drive) memberikan nomor sektor absolut atau fisik (relatif terhadap perangkat fisik lengkap daripada partisi tertentu). Saat Anda menjalankan pemindaian permukaan, klik pada blok tertentu, itu akan menampilkan presentasi heksadesimal dari sektor pertama dari blok itu, kemudian Anda dapat pergi ke sektor lain dengan mengklik "Pergi ke" (Anda juga harus menentukan absolut nomor sektor), dan jika Anda mengklik kotak "Deteksi informasi file untuk sektor", ini akan menampilkan file mana yang menempati sektor itu, dan daftar lengkap sektor yang ditempati oleh file yang sama. Ada peringatan, meskipun: ia akan mencoba untuk benar-benar membaca setiap sektor yang diminta dengan cara itu (untuk menampilkan isinya), yang dapat secara tidak perlu menekankan area yang sudah rusak dan memperluasnya, tanpa memulihkan apa pun seperti yang coba dilakukan oleh perangkat lunak kloning. (Saya mengalami masalah serupa yang harus saya tangani sekarang juga ; dalam situasi saya, jumlah sektor buruk meningkat saat saya mencoba metode ini, meskipun saya mencoba mengidentifikasi file yang terpengaruh dengan mengakses sektor terdekat alih-alih langsung meminta yang sudah diidentifikasi sebagai buruk).

- Recuva menyediakan daftar nomor cluster logis untuk setiap file di tab "Info". Itu tidak melakukan secara langsung apa yang Anda butuhkan di sini: ini menyediakan daftar cluster yang ditempati oleh file tertentu, tetapi tidak memungkinkan untuk menemukan file mana yang ada di sektor apa. Namun, fungsionalitas Recuva yang kurang dikenal ini dapat digunakan bersama dengan alat di atas: setelah Anda mengetahui file mana yang menempati bad sector tertentu, akan berguna untuk mengetahui sektor mana yang juga ditempati oleh file itu, untuk melihat apakah itu direkam berurutan atau sangat terfragmentasi misalnya (dalam kasus saya beberapa file yang terpengaruh sangat terfragmentasi, jadi saya tidak dapat mengekstraknya secara praktis dari pemulihan ddrescue parsial dari berbagai sektor yang relevan tanpa informasi sistem file, seperti yang saya maksudkan pertama kali).
Pastikan untuk menjalankan file scan cepat (metode default), itu cukup untuk mengurai MFT dan mendapatkan indeks dari file yang saat ini dialokasikan, dan itu tidak akan membebani hard drive dengan menguraikannya sepenuhnya jika area yang rusak terletak lebih jauh (seperti halnya dengan milik saya: semua sektor buruk berada di sekitar tanda 2TB). Jika file yang berisi bad sector cukup besar dan bad sector tidak tepat di awal, itu seharusnya tidak berbahaya, tetapi dengan file kecil atau jika bad sector terjadi di awal, cukup klik di atasnya untuk mendapatkan daftar cluster yang mereka tempati dapat meningkatkan kerusakan karena Recuva akan mencoba menampilkan pratinjau atau setidaknya header.

- Defraggler dari Piriform (perusahaan yang sama yang mengembangkan Recuva): saat mengklik blok tertentu di peta volume, itu akan menampilkan daftar file yang terdapat dalam blok yang sama (bahkan file yang tidak terfragmentasi). Dan saat mengklik nama file di tab "Daftar file" (yang hanya menampilkan file yang terfragmentasi), ini akan menyorot semua blok yang berisi setidaknya satu sektor milik file itu. Sayangnya tidak ada indikasi numerik interval offset / sektor / cluster. (Saya menulisnya untuk meminta beberapa penyempurnaan yang akan membuat fitur hebat ini lebih praktis.)

- R-Studio (juga tidak gratis tetapi ini adalah salah satu perangkat lunak pemulihan terbaik yang pernah saya coba) dapat menampilkan sektor yang ditempati oleh file apa pun di pohon pemulihan (bahkan yang ada di "File yang ditemukan ekstra" yang saat ini tidak dialokasikan tetapi diukir sesuai dengan struktur header / footer mereka) dengan membuka penampil heksadesimal dan mengklik tab "Sektor" - tetapi saat ini tidak ada cara untuk mengekspor daftar tersebut (Anda dapat menyalin nilai satu per satu dengan CTRL + C, tetapi tidak memilih semuanya ; dengan Recuva atau HD Sentinel dimungkinkan untuk menyalin seluruh daftar untuk penyelidikan lebih lanjut).
Anda juga dapat mengidentifikasi file mana yang ada di sektor tertentu dengan metode ini:
1) Dari pohon pemulihan, klik File => "Tampilkan file di HexEditor".
2) Kemudian kembali ke tab tampilan perangkat, klik kanan pada volume yang Anda periksa, lalu "Lihat / Edit": ini akan membuka HexEditor (yang lebih merupakan penampil heksadesimal, secara default dalam mode hanya-baca, sebagaimana mestinya untuk tujuan pemulihan data). Di sini Anda dapat mengetik nomor sektor dan menampilkannya serta mendapatkan nama file miliknya. Namun ada ketidakkonsistenan: untuk menampilkan konten sektor Anda harus mengetikkannya logis nomor, tetapi untuk mendapatkan nama file yang dimilikinya, Anda harus mengetiknya mutlak nomor, menambah atau menghapus offset partisi.
Sekali lagi, dalam kasus HDD yang gagal (kegagalan perangkat keras), metode ini berisiko karena perangkat lunak akan mencoba mengakses sektor yang rusak.

- Dengan ddru_ntfsfindbad, alat dari set alat ddr_utility, Anda bisa mendapatkan daftar file yang menempati sektor yang tidak dapat dibaca oleh ddrescue (program kloning baris perintah yang dirancang khusus untuk memulihkan data dari unit penyimpanan yang gagal). Ia bekerja dengan file log yang dihasilkan oleh ddrescue. Dalam hal ini Anda sudah mengkloning HDD yang gagal dengan Clonezilla, tetapi jika hanya ada beberapa bad sector, masih memungkinkan untuk membuat file log kustom dari template, tanpa harus melalui proses kloning lagi. Ini memiliki keuntungan dalam memberikan informasi rinci, terutama ukuran data yang rusak untuk setiap file yang terpengaruh. Ini berjalan di Linux, yang merupakan peringatan lain, jadi ini bukan solusi termudah dalam kasus Anda, tetapi mungkin membantu orang lain yang benar-benar melakukan pemulihan menggunakan ddrescue.
