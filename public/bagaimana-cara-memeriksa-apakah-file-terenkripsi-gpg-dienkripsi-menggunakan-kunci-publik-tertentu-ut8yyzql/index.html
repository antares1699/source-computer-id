Bagaimana cara memeriksa apakah file terenkripsi GPG dienkripsi menggunakan kunci publik tertentu?

Pertimbangkan jika file dienkripsi menggunakan perintah

misalnya

gpg --output test.txt.gpg --encrypt --recipient test@example.com test.txt

Sekarang, pertimbangkan orang lain menerima test.txt.gpg kunci pub test@example.com, bagaimana melakukan pemeriksaan untuk memastikan file tersebut benar-benar dienkripsi menggunakan kunci pub?

Beberapa perilaku telah berubah di versi yang lebih baru dari gpg (Saya sedang menguji pada gpg v2.2.4). Ini meminta Anda memasukkan kata sandi saat melakukan --list-packets. Dan sekarang secara default, -k tidak lagi menampilkan ID kunci, hanya sidik jari 40 karakter.

Berikut beberapa perintah yang akan melewati perintah dan menampilkan ID kunci:

# show public key ID that it was encrypted with, skipping prompts
gpg --pinentry-mode cancel --list-packets file.gpg

# list keys with the key IDs
gpg -k --keyid-format long

Anda kemudian dapat mencocokkan ID kunci 16 karakter dari --list-packets ke daftar dari -k.

Anda bahkan tidak perlu memiliki kunci publik (cukup ketahui ID itu). Dengan asumsi bahwa seseorang tidak sengaja menyembunyikan kunci yang digunakan untuk mengenkripsi file / pesan, maka Anda dapat menggunakan perintah list-packets untuk melakukan apa yang Anda minta.

gpg --list-packets test.txt.gpg

Itu akan menunjukkan kepada Anda daftar setiap paket dekripsi yang terdaftar di header pesan / file PGP-ed. Ada cara untuk mengenkripsi ke kunci dan menyembunyikan kunci itu dari daftar ini, tetapi menggunakan perintah normal (seperti yang ada di contoh Anda) akan menghasilkan semua kunci dekripsi yang terdaftar, seperti ini.

gpg --list-packets crypto-text.pgp
# off=0 ctb=85 tag=1 hlen=3 plen=526
:pubkey enc packet: version 3, algo 16, keyid 2206D60BA555DCB0
        data: [2045 bits]
        data: [2047 bits]
# off=529 ctb=85 tag=1 hlen=3 plen=526
:pubkey enc packet: version 3, algo 16, keyid 0D6229B307ED0210
        data: [2043 bits]
        data: [2048 bits]
# off=1058 ctb=85 tag=1 hlen=3 plen=526
:pubkey enc packet: version 3, algo 16, keyid 0E9AFA6C61A4DC66
        data: [2048 bits]
        data: [2046 bits]

Perhatikan tiga ID Kunci berbeda yang terdaftar. Ya, Anda dapat menebaknya, masing-masing kunci tersebut dapat digunakan untuk mendekripsi pesan khusus ini.

Poin bonus jika Anda memiliki kunci publik karena perintah --list-packets juga akan memberi tahu Anda nama ramah untuk masing-masing paket tersebut di header pesan. (Dan Anda bisa menggunakan kunci publik yang sudah Anda miliki).

Tapi, kecuali seseorang dengan sengaja menyembunyikan kunci dekripsi dari daftar di header pesan , Anda dapat melihat semua kunci PGP yang dapat digunakan untuk mendekripsi pesan tanpa memiliki akses ke kunci publik atau privat untuk salah satunya.

Sebenarnya saya menggunakan ini jauh lebih banyak yang ingin saya akui, biasanya ketika orang-orang di perusahaan saya bertukar pesan dengan PGP dan tiba-tiba seseorang tidak dapat membukanya. Mereka selalu menyalahkan implementasi perangkat lunak. Sejauh ini, setiap saat, pesan telah disusun, dienkripsi, dan setelah enkripsi, penerima baru ditambahkan ke daftar TO. Saya dapat menyalin teks sandi ke file teks-kripto.pgp (seperti dalam contoh saya) dan melihat dengan tepat kepada siapa pesan itu dienkripsi; menunjukkan kepada mereka bahwa cenderung menghentikan permainan "plugin perangkat lunak Anda rusak" dengan kuat di jalurnya.

Ini hanya mungkin jika Anda adalah pemilik penerima pribadi kunci. Jika Anda tidak dapat mendekripsi pesan dengan kunci privat, itu tidak dienkripsi dengan kunci publik yang cocok (atau telah dimodifikasi).

Tetapi kemampuan membalikkan enkripsi hanyalah (meskipun kuat) indikasi , bukan bukti. Sebagai ujian sejati untuk pencipta (dan integritas), penandatanganan digunakan, lihat sini . Perlu diketahui bahwa sangat mungkin untuk menandatangani dan mengenkripsi dengan kunci yang berbeda. Dan ada diskusi jika salah satu sebagai pengirim harus menandatangani terlebih dahulu, kemudian mengenkripsi, atau sebaliknya.
