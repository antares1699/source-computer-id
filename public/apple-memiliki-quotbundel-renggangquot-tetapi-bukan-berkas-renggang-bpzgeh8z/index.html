Apple memiliki "bundel renggang" tetapi bukan berkas renggang?

Backstory:

Saya menggunakan Mac (10.6.4), dan sampai hari ini ketika saya mengunduh torrent menggunakan Vuze, tidak peduli konfigurasi apa yang saya coba, Vuze mengalokasikan ruang untuk seluruh file sebelum mulai mengunduh file tersebut. Untuk waktu yang lama, google yang cepat membuat saya percaya itu adalah bug Vuze, tetapi setelah bertahun-tahun berlalu dan tidak ada yang berubah, pencarian yang lebih rinci memberi tahu saya bahwa itu karena HFS + tidak mendukung file yang jarang.

Pertanyaan:

File renggang wiki mengatakan bahwa HFS + Apple tidak mendukung file renggang, dan Apple Time Machine dan FileVault menggunakan "gambar renggang" dan "bundel renggang".

Saya mengerti apa itu sparse bundle (image dibagi menjadi 8Meg chunks) tapi selain itu, apakah ada perbedaan antara "sparse images" dan hanya file sparse biasa? Bagaimana HFS + tidak mendukung file sparse namun masih mendukung gambar sparse dan bundel sparse?

Itu karena ketersebaran ditangani pada tingkat abstraksi yang berbeda untuk gambar renggang vs. file renggang. Jika Anda menyimpan file "langsung" di disk, sistem file (mis. HFS +) menangani pemetaan urutan byte yang menyusun file menjadi urutan byte pada disk. File sparse adalah file yang memiliki celah dalam urutan byte-nya, dan sistem file yang mendukung ini akan melewatkan byte yang hilang tersebut ketika memetakan file ke disk.

Saat Anda menggunakan image disk, pemetaannya sedikit lebih rumit: Anda memiliki byte dalam file yang sedang dipetakan (oleh sistem file bagian dalam) menjadi byte dalam disk (virtual). Ini kemudian dipetakan (oleh image mounter) menjadi byte dalam image disk (file), yang kemudian dipetakan (oleh sistem file luar) menjadi byte pada disk sebenarnya. Image mounter menangani pemetaan yang jarang (dan AIUI hanya di antara / di luar file, bukan di dalamnya), tetapi itu tidak berarti bahwa lapisan lain (sistem file) dapat melakukannya. Sejauh menyangkut filesystem luar, file image hanyalah urutan byte (non-sparse).

BTW, image mounter juga dapat menambahkan fitur lain yang tidak ditangani oleh filesystem saja. Misalnya, image disk Mac telah mendukung kompresi secara kasar selamanya, tetapi dukungan asli untuk kompresi tidak ditambahkan ke HFS + hingga OS X v10.6.

Penjelasan Gordon Davisson mencerahkan: ini adalah bagaimana saya menggunakan gambar-gambar yang tersebar sebagai direktori unduhan torrents.

Pertama-tama buat gambar (saya atur ke 50GB, tetapi ukuran apa pun akan berfungsi)

hdiutil create -size 50g -type SPARSEBUNDLE -nospotlight -volname "Torrents repository" -fs "Case-sensitive Journaled HFS+" Torrents.sparsebundle

Kemudian klik dua kali gambar Torrents.sparsebundle yang baru saja dibuat untuk memasangnya.

Setup klien torrent favorit Anda untuk menggunakan volume mount yang disebut "Torrents repository" sebagai repositori sementara, saya sarankan untuk mengaturnya sehingga file yang sudah selesai akan dipindahkan dari gambar (ke folder Downloads misalnya).

Ia bekerja dengan sangat baik, terutama bila Anda hanya memilih beberapa file dalam torrent yang besar.

Saya juga menambahkan skrip startup ke sistem saya:

hdiutil compact Torrents.sparsebundle 
hdiutil attach Torrents.sparsebundle

Perintah pertama akan mengambil kembali ruang disk yang tidak digunakan, sedangkan perintah kedua akan secara otomatis memasang gambar. Sayangnya, pengoperasian ringkas tidak dapat dilakukan saat gambar dipasang, jadi menurut saya peluang terbaik adalah saat memulai sistem sebelum memasangnya.
