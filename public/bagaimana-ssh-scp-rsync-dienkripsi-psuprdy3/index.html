Kunci sesi simetris yang dibuat secara acak digunakan untuk mengenkripsi komunikasi antara klien dan server. Selama koneksi kunci dibagi oleh kedua belah pihak, ketika sesi berakhir, kunci tersebut dimusnahkan.

SSH versi 1 (SSH-1) hanya menggunakan satu kunci, sedangkan SSH-2 memiliki beberapa: masing-masing arah (klien-> server, server-> klien) dan lainnya untuk pemeriksaan integritas.

SSH-2 menyediakan cara bagi salah satu sisi koneksi SSH untuk memulai penguncian ulang ("Sesi rekeying"), menyebabkan klien dan server menegosiasikan kunci sesi baru.

Sumber: O'REILLY - SSH The Definitive Guide

Seperti yang disebutkan EightBitTony, gunakan scp.

rsync mungkin tidak dienkripsi. Saya t biasanya berjalan di atas ssh secara default (dan karenanya dienkripsi) jika tidak berbicara dengan server rsync, tetapi mungkin telah dikonfigurasi / dikompilasi untuk menggunakan rsh atau shell jarak jauh lainnya.

ssh (dan scp, yang menjalankan SSH) dienkripsi menggunakan enkripsi kunci publik / pribadi, tidak ada yang berjalan melalui koneksi yang dapat diinterpretasikan tanpa akses ke kunci privat. Orang-orang dicegah tidak hanya membaca konten file, tetapi dari mengetahui Anda bahkan mentransfernya, mereka hanya dapat melihat lalu lintas terenkripsi.

Yang pasti, Anda bisa menggunakan,

rsync -e ssh .....................
