Windows 10 menyertakan server OpenSSH sebagai fitur standar.

Untuk mengaktifkan server OpenSSH:

  • Buka Pengaturan> Aplikasi> Aplikasi & fitur
  • Di sebelah kanan, klik Fitur pilihan
  • Di jendela Fitur opsional, klik Tambahkan fitur
  • Dalam daftar fitur, cari "OpenSSH Server". Fitur tersebut dapat lebih mudah ditemukan dengan mengetikkan "buka" di kolom pilihan (ini cukup untuk menemukan fiturnya).
  • Klik pada entri tersebut dan kemudian pada tombol "Instal"
  • Tunggu hingga selesai dan mulai ulang Windows 10.

Ini akan menginstal perangkat lunak OpenSSH Server di folder tersebut C:\Windows\System32\OpenSSH, yang akan berisi aplikasi klien SSH dan alat server berikut: sftp-server.exe, ssh-agent.exe, ssh-keygen.exe, sshd.exe dan file konfigurasi sshd_config.

Server SSH dikonfigurasi untuk dijalankan sebagai layanan bernama sshd. Jika tidak dimulai secara otomatis, Anda perlu mengkonfigurasinya secara manual.

Kuncinya adalah saya ingin terus menggunakan mesin sementara orang tersebut menjalankan beban komputasi di dalamnya

Ini mungkin, tetapi beban komputasi tersebut akan memperlambat mesin. Saya pasti akan mengatur prioritas proses yang menjalankan beban komputasi ke Di Bawah Normal jika Anda ingin memprioritaskan kelancaran operasi Anda. Ini berarti semua proses yang berjalan default akan diprioritaskan untuk waktu CPU yang tersedia sebelum komputasi dimuat, tetapi cara lain untuk melihatnya adalah jika ada BANYAK proses lain yang berjalan pada mesin yang sama dengan prioritas Normal, itu juga menggunakan hingga 100 % dari CPU, beban komputasi akan mendapatkan sedikit atau tanpa waktu CPU. Ini mungkin diinginkan oleh Anda, dan kemungkinan bahwa Anda menggunakan CPU pada 100% (atau memiliki banyak proses yang berjalan melakukannya untuk Anda) kemungkinan besar rendah.

Sejauh yang saya tahu, persyaratan itu mengecualikan semua bentuk RDP / VNC, tetapi saya tidak yakin tentang itu (terutama RDP): bisakah Anda konfirmasi / tolak?

Itu betul. RDP dari Windows versi Starter / Student / Home / Pro memiliki batasan yang disengaja hanya untuk satu login pada satu waktu (pengecualian di Enterprise). ada beberapa super cara teduh untuk memecahkannya sehingga tidak akan mem-boot Anda keluar ketika orang lain masuk dari jarak jauh menggunakan RDP, tetapi saya yakin mendiskusikan hal-hal seperti itu tidak diizinkan di SuperUser dan percakapan itu mati di sini.

VNC / NX / Berbagi layar semua akan mengambil alih sesi Anda yang saat ini masuk dan menggunakannya dari jarak jauh, jadi jelas tidak boleh.

Saya berpikir untuk menjalankan server SSH. Apakah itu akan berhasil?

Iya! Ini pasti akan berhasil, dan Windows 10 memiliki SSH bawaan secara default. Anda dapat mengaktifkannya menggunakan langkah-langkah berikut:

  1. Setelan> Aplikasi dan klik "Kelola fitur opsional" di bawah Aplikasi & fitur.

  2. Klik "Tambahkan fitur" di bagian atas daftar fitur yang diinstal. Jika Anda sudah menginstal klien SSH, itu akan muncul di daftar di sini.

  3. Gulir ke bawah, klik opsi "OpenSSH Server", dan klik "Instal".

Orang tersebut harus dapat menggunakan SSH dalam menggunakan Akun Pengguna, yang ingin Anda siapkan untuk mereka, dan mengeluarkan perintah baris perintah untuk menjalankan proses saat Anda masuk secara fisik ke mesin. Biola!

Jawaban lainnya semuanya benar (pada saat penulisan ini). Namun, saya ingin memberikan alternatif untuk server Windows SSH bawaan untuk Anda perhatikan:

Anda mungkin ingin menginstal Cygwin , yang telah ada selama sekitar dua dekade, didukung dengan baik dan terus diperbarui, dan IMHO sangat stabil. Ini memiliki OpenSSH disertakan, dan Anda dapat dengan mudah menginstal server Cygwin OpenSSH sebagai layanan. Saya menggunakan pengaturan seperti itu selama bertahun-tahun dan memulainya jauh sebelum Microsoft berpikir untuk mengintegrasikan subsistem Linux atau SSH ke dalam Windows.

Jika Anda belum pernah mendengarnya: Cygwin terutama adalah DLL yang menyediakan sebagian besar API Linux / Unix / Posix di bawah Windows sehingga sebagian besar utilitas userland Linux dapat di-porting ke Windows. Nomenklaturnya agak mengkhawatirkan, karena Cygwin juga biasa digunakan sebagai nama "distribusi" Linux, yaitu perangkat lunak Linux yang telah di-porting ke Windows dengan bantuan Cygwin DLL dan yang ditawarkan di Situs Cygwin juga.

Keuntungan dari Cygwin adalah menyediakan lingkungan Linux, termasuk C toolchain, bahasa scripting yang umum, bash dll. Itu tergantung pada latar belakang teman Anda jika ini merupakan keuntungan. Dalam setiap kasus, keuntungan yang dihasilkan adalah Anda dapat mengkonfigurasi server OpenSSH persis seperti di Linux - ada banyak tutorial yang membahas subjek itu.

Kerugian dari Cygwin adalah bahwa server OpenSSH-nya (tentu saja) tidak terintegrasi secara mendalam ke dalam Windows seperti server SSH bawaan.

Mengenai metode akses: Server OpenSSH secara teratur ditambal dan IMHO sangat aman, jadi akan aman ( dalam teori ) untuk mengkonfigurasi router Anda dengan penerusan port sehingga teman Anda dapat terhubung langsung ke PC Anda melalui SSH. Namun, fakta bahwa ini dianggap gaya yang buruk ada alasannya. Oleh karena itu, saya merekomendasikan untuk mengkonfigurasi VPN antara teman Anda dan router Anda, menggunakan kriptografi terkuat yang tersedia di kedua ujungnya, dan menjalankan sesi SSH melalui VPN.

Kata terakhir tentang TeamViewer (karena bounty Anda dan beberapa komentar menyebutkannya):

Layanan ini dijalankan oleh perusahaan Jerman yang saya percayai secara pribadi; mereka harus mematuhi undang-undang perlindungan data yang ketat dan akan langsung hancur jika ada tersangka sekecil apa pun atau bahkan bukti bahwa mereka merekam atau bahkan mendekripsi sesi. Namun, karena sifat layanan mereka, semua lalu lintas yang melewati sesi TeamViewer dirutekan melalui server mereka, sehingga sesi sniffing secara teoritis dimungkinkan bagi mereka (yang, omong-omong, berlaku untuk semua layanan / perusahaan serupa yang bersaing. demikian juga).

TeamViewer telah meyakinkan beberapa kali bahwa sesi dienkripsi ujung ke ujung, yang berarti bahwa lalu lintas melewati server mereka, tetapi mereka secara teknis bahkan tidak dapat mendekripsi, bahkan jika mereka ingin melakukannya. Tetapi sekali lagi, sementara saya pribadi mempercayai mereka, selama perangkat lunak mereka bukan open source, tidak ada yang dapat memverifikasi pernyataan ini, atau apakah mereka menerapkan enkripsi dengan benar.

Terakhir, seperti yang telah ditunjukkan orang lain, teman Anda tidak akan dapat menjalankan sesi TeamViewer-nya sendiri tanpa mengganggu Anda. TeamViewer lebih merupakan solusi berbagi layar: Pihak lain dapat melihat apa yang Anda lakukan, dan sebaliknya.

Oleh karena itu, saya selalu lebih suka SSH atau VPN (berdasarkan perangkat lunak sumber terbuka) daripada TeamViewer dan layanan serupa bila memungkinkan.
