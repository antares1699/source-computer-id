Mengapa MacOSX mencoba menyelesaikan alamat IP tanpa bantalan sebagai nama host?

Yang ini menyebabkan seorang teman sakit kepala ketika mengatur koneksi internet yang dialihkan yang tidak memiliki alamat IP empuk di kertas konfigurasi seperti:

192.168.019.254

Tidak peduli bagaimana dia mencoba mengonfigurasi tumpukan jaringan Mac, itu tidak berhasil. Tetapi GUI juga tidak mengeluh tentang kesalahan.

Saya kemudian menemukan bahwa melakukan ping ke alamat nol empuk ini di terminal menunjukkan "tidak dapat menyelesaikan" sementara melakukan ping ke alamat yang sama tanpa nol empuk (192.168.19.254 vs 192.168.019.254) menunjukkan "tidak dapat menjangkau" yang membawa banyak cahaya ke dalam misteri ini. Menghapus angka nol yang tidak berguna membuat semuanya bekerja.

Apakah ini bug atau fitur? Saya tidak pernah mengalami perilaku aneh seperti itu sebelumnya - tidak peduli apakah saya menggunakan GUI atau CLI untuk mengkonfigurasi atau menguji hal-hal IP.

Terjemahan dari string ke alamat biasanya dilakukan oleh fungsi POSIX getaddrinfo() . Fungsi ini pertama kali memeriksa alamat IP numerik menggunakan inet_addr() , dan jika gagal maka akan mencoba untuk menyelesaikan string sebagai nama domain. inet_addr() menginterpretasikan angka dengan awalan 0 sebagai oktal, jadi misalnya 010 akan menjadi 8, dan 019 akan menjadi kesalahan (dan karena itu akan diselesaikan sebagai nama domain). Perilaku yang sama terjadi di Linux dan Solaris juga.

Dari getaddrinfo() :

Jika kelompok alamat yang ditentukan adalah AF_INET atau AF_UNSPEC, string alamat menggunakan notasi titik standar Internet seperti yang ditentukan pada inet_addr() adalah valid.

Dari inet_addr() :

Semua angka yang diberikan sebagai bagian dalam notasi desimal bertitik IPv4 dapat berupa desimal, oktal, atau heksadesimal, seperti yang ditentukan dalam standar ISO C (yaitu, 0x atau 0X di depan berarti heksadesimal; jika tidak, '0' di depan berarti oktal; jika tidak, nomor tersebut diartikan sebagai desimal).

Catat alamat 192.168.8.254 dalam tanda kurung:

$ ping 192.168.010.254
PING 192.168.010.254 (192.168.8.254): 56 data bytes
Request timeout for icmp_seq 0
