Mengapa "Opsi Pengindeksan" Pencarian Windows memilih drive yang salah jika dikloning?

Saya mengkloning HDD Windows 10 ke SSD Samsung menggunakan "Perangkat Lunak Migrasi Data Samsung", lalu menginstal salinan baru Windows 10 pada SSD. SSD sekarang C: dan HDD D:.

Ketika di Control Panel> Indexing Options> Modify, folder yang saya tambahkan untuk C:\ sebenarnya memilih D:\. Sebagai contoh, saya akan memilih C:\Windows> OK> lalu Modify lagi dan D:\Windows yang dipilih; atau jika saya memilih folder yang tidak ada di D:, seperti C:\util, itu menambahkan entri seperti D:\util\ (unavailable).

Ada sesuatu yang membingungkan Opsi Pencarian dan Pengindeksan Windows karena drive kloning. Saat mencari di registri, saya dapat melihat entri seperti:

HKLM\SOFTWARE\Microsoft\Windows Search\CrawlScopeManager\Windows\SystemIndex\DefaultRules\12
  • REG_SZ
    Nama: URL
    Data: file:///C:\\[9a6b2440-0cb7-4d60-a957-7a1682cf61c4]\\WINDOWS\\

Mungkin GUID setelah C:\\ yang membingungkan Opsi Pengindeksan, tetapi GUID ini tidak muncul di mana pun di registri selain di bawah Windows Search. Bukan pengenal unik volume yang dapat dilihat menggunakan Mountvol, dan bukan pengenal GPT (atau jenis partisi) yang dapat dilihat menggunakan DiskPart.


Catatan: orang lain mengalami masalah yang sama (di Windows 8 juga), lihat:

Pertama-tama, saya dapat mereproduksi masalah Anda:

Masalahnya disebabkan oleh file sistem \System Volume Information\IndexerVolumeGuid di setiap volume kloning. Rupanya Anda benar tentang fakta bahwa sistem dibingungkan oleh GUID yang Anda lihat di entri registri di bawah HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Search\CrawlScopeManager\Windows\SystemIndex\*, dan file ini seharusnya merupakan penyedia GUID.

Karena volume dikloning, file khusus ini identik di antara masing-masing seperti file lainnya. Dan TIDAK akan "diperbarui" (yaitu dibuat ulang untuk menghindari konflik) saat volume dipasang, seperti serial volume:

Namun, tampaknya file ini aman untuk dihapus, dan setelah volume dipasang kembali, yang baru akan dibuat; masalah akan hilang setelah SEMUA volume yang memiliki GUID duplikat dipasang kembali dengan GUID unik (jadi reboot mungkin diperlukan jika melibatkan volume sistem saat ini):

  • volume unique id (mountvol X: /L) tidak ada hubungannya dengan filesystem, tetapi hanya partisi (tabel). Untuk disk GPT, Windows menggunakan unique partition GUID sebagai volume unique id. Untuk disk MBR, gunakan pengenal disk (uniqueid pada diskpart) dan posisi awal partisi untuk membentuk volume unique id. FAT32 dan NTFS memiliki volume serial; itu hanya salah satu dari NTFS yang secara teknis lebih panjang (fsutil fsinfo ntfsinfo X:, tetapi jika Anda memeriksa dengan dir X: hanya sebagian dari serial NTFS yang ditampilkan, mungkin untuk kompatibilitas dengan serial FAT32 yang lebih pendek).
