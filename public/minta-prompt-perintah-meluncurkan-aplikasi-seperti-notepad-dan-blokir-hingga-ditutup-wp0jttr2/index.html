Minta prompt perintah meluncurkan aplikasi seperti Notepad dan blokir hingga ditutup

Saya memiliki skrip Batch pada dasarnya saya ingin menampilkan jendela notepad kepada pengguna yang dibuka dengan file konfigurasi untuk diedit pengguna setiap kali mereka menjalankan skrip, dan saya ingin Command Prompt menunggu hingga program ini ditutup untuk melanjutkan eksekusi, dan jalankan program yang menggunakan file konfigurasi ini.

Saya ingin melakukan ini karena saya tidak ingin mengambil input pengguna dalam skrip Batch dengan gaya perintah "set / P", dan harus menyiapkan pembacaan dan penulisan parsing ke file konfigurasi yang memiliki banyak nilai untuk diperiksa.

Saya percaya perilaku standar pada Windows saat menjalankan program yang dapat dieksekusi, untuk file "exe" itu akan membuka program dalam prosesnya sendiri dan segera mengontrol kembali ke skrip / prompt perintah. Untuk file skrip "bat" dan "cmd", ia akan menjalankan skrip dan menunggu hingga kontrol dikembalikan ke skrip pertama atau prompt perintah. Ini harus menjadi perilaku ketika setidaknya memanggil skrip lain dengan perintah "CALL" atau memanggilnya secara langsung.

Sekarang di Linux saya percaya perilaku default untuk konten yang dapat dieksekusi yang merupakan kode mesin asli yang dapat datang tanpa ekstensi, dan skrip seperti file "sh", dijalankan dengan cara memblokir, kecuali Anda menentukan karakter ampersand ("&") di suatu tempat di jalur masukan.

Jadi saya mencoba untuk mencapai perilaku pemblokiran pada Windows karena lebih mudah dilakukan di Linux, dan ingin penutupan program untuk memicu skrip asli untuk melanjutkan, atau mengembalikan kontrol ke prompt perintah.

Anda bisa menggunakan keduanya start perintah (perhatikan urutan /B /WAIT sakelar)

start "" /B /WAIT notepad.exe "somefile"

atau sederhananya

notepad.exe "somefile"

Sesuai tautan di atas:

Saat Anda menjalankan aplikasi antarmuka pengguna grafis 32-bit (GUI), cmd tidak menunggu aplikasi ditutup sebelum kembali ke prompt perintah. Perilaku ini tidak terjadi jika Anda menjalankan file aplikasi dari skrip perintah .

Sunting sehubungan dengan komentar Karan: Mengapa / B diperlukan?

/B memulai aplikasi tanpa membuka Command Prompt baru jendela.

  • Sejujurnya: jika Anda menjalankan file aplikasi antarmuka pengguna grafis kemudian tidak Jendela Command Prompt baru terbuka meskipun tombol /B dihilangkan.
  • Di sisi lain, jika Anda menjalankan file alat baris perintah , menghilangkan saklar /B menyebabkan menjalankan alat itu di jendela Command Prompt baru yang akan segera ditutup setelah itu berakhir dan semua hasilnya hilang .

Oleh karena itu, saya menggunakan sakelar /B /WAIT bersama-sama sebagai masalah prinsip umum: pertahankan perilaku yang sama terlepas dari disposisi program yang dimulai dan jangan memikirkannya ...

Suplemen atas pesanan /B /WAIT sakelar dengan contoh dan lebih banyak tautan dari StackOverflow : start /WAIT /B tidak berfungsi (argumen /wait diabaikan) sementara start /B /WAIT berfungsi ...

notepad.exe | echo > NUL

Apa yang dilakukannya adalah, ini membuka notepad.exe dan semua outputnya diarahkan ke perintah echo > NUL. Untuk mengarahkan ulang seluruh output, program harus selesai. Oleh karena itu kita harus menunggu sampai notepad.exe selesai. Outputnya bergema. Echo tidak menggunakan input dengan cara apa pun dan hanya mengeluarkan sesuatu ke NUL. NUL adalah aliran jendela standar yang tidak mengarah ke mana pun.

Saat menulis pertanyaan, satu jawaban datang kepada saya, meskipun tidak mencapai hasil yang saya inginkan.

Ini akan membuka program apa pun yang ingin saya buka, dan menambahkan perintah "jeda" langsung setelah itu pada baris di bawah. Ini akan menghentikan eksekusi skrip cukup sampai pengguna menutup jendela program baru yang baru saja muncul, dan perlu menyadari bahwa mereka perlu memajukan skrip secara manual dari titik itu. Jeda mencetak prompt yang menjelaskan apa yang terjadi untuk kemungkinan memberi tahu pengguna ("Press any key to continue . . .").

Ini bukan masalah besar kecuali skrip diminimalkan karena alasan apa pun, dan pengguna yakin itu telah selesai. Ini juga tidak bagus karena tidak secara otomatis melanjutkan menjalankan sisa skrip saat program ditutup, yang merupakan efek yang saya inginkan.
