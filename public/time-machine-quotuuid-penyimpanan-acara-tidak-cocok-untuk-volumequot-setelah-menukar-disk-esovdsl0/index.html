Time Machine "UUID penyimpanan acara tidak cocok untuk volume" setelah menukar disk

Hard drive baru saya mati minggu lalu dan harus menyimpan cadangan drive lama saya ke Mac Mini saya, yang menjalankan Snow Leopard. Saya kemudian dapat memulihkan cadangan Time Machine terbaru saya.

Ketika saya meningkatkan beberapa bulan yang lalu saya menggunakan Carbon Copy dan saya memiliki masalah izin.

Jadi saya memiliki drive lama saya di sistem saya saat ini, tetapi ketika saya mencoba melakukan backup Time Machine, itu SANGAT lambat. Ini menggunakan pengaturan / lokasi yang sama seperti sebelumnya. saya mengunduh TM Buddy , yang mengatakan...

Starting standard backup
Backing up to: /Volumes/Mac Time Machine/Backups.backupdb
Event store UUIDs don't match for volume: Macintosh HD
Waiting for index to be ready (100)
Waiting for index to be ready (100)
Node requires deep traversal:/ reason:must scan subdirs|new event db|
No pre-backup thinning needed: 109.39 GB requested 
      (including padding), 121.15 GB available

Saya mencoba melakukan pencadangan agar saya dapat memasukkan drive baru lainnya, sehingga saya dapat melakukan pemulihan Time Machine, seperti yang saya lakukan minggu lalu.

Apa yang dapat saya lakukan untuk memperbaiki masalah ini?

Untuk mengatasinya: tunggu.

  • Setelah melakukan pemulihan penuh, Time Machine akan selalu membuat cadangan lengkap, sesuai desain. Tanpa mengetahui Mengapa Apple menganggap ini diperlukan, saya lebih memilih cadangan yang andal dari waktu ke waktu dan ruang disk. Lihat juga milik Apple Mac OS X 10.5: Time Machine melakukan pencadangan penuh setelah pemulihan penuh .

  • Dalam semua kasus lainnya: Time Machine telah mendeteksi bahwa ia tidak dapat mengetahui apa yang ada di cadangan Anda, dan apa yang tidak, dan perlu membandingkan keduanya. Anda mungkin juga melihat Node requires deep traversal.

Ini tidak terkait dengan ID disk (perangkat keras) itu sendiri. TM menyimpan FSEvents ID itu digunakan untuk cadangan terakhir di "atribut diperpanjang" com.apple.backupd.SnapshotVolumeLastFSEventID pada disk. Biasanya, yang diperlukan untuk menentukan apa yang telah berubah adalah membandingkan nilai itu dengan ID yang dikenal oleh OS X. Namun, jika karena alasan tertentu database OS X FSEvents tidak dapat lagi dipercaya, ia akan membuat yang baru, yang mengubah keunikannya. UUID. TM memeriksa untuk melihat apakah database FSEvents dapat digunakan untuk disk cadangan tertentu dengan membandingkan UUID unik tersebut dengan UUID yang disimpan dengan cadangan, di com.apple.backupd.SnapshotVolumeFSEventStoreUUID. Jadi setelah database FSEvents baru dibuat, UUID ini tidak lagi cocok dan TM perlu membandingkan harddisk dengan cadangannya, atau mungkin perlu membuat cadangan lengkap.
