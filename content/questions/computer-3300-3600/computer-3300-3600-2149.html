Jawaban singkat - Saya tidak akan repot, karena itu tidak terlalu berharga. Bergantung pada server DNS Anda - pencarian dilakukan relatif cepat, terutama jika Anda menggunakan yang didedikasikan oleh ISP Anda (latensi bijaksana). Jadi, upaya untuk melakukan ini tidak sebanding dengan keuntungan minimal - hanya jika Anda benar-benar mengalami perlambatan dari layanan DNS yang dialokasikan atau pihak ketiga.

\n\n

Lihat ini (jika di Windows):

\n\n
\n

ipconfig / displaydns

\n
\n\n

Ini harus mengekspos entri cache dari resolusi DNS, masing-masing dengan miliknya sendiri TTL .

\n\n

Mengedit tuan rumah file, bagaimanapun, akan menambah resolusi statis. Saya, secara pribadi, saya hanya akan menambahkan beberapa pemetaan di sana (yang saya yakin) dan hanya itu. Jika dilakukan secara manual - Anda selalu berisiko mendapatkan entri yang sudah ketinggalan zaman. Namun, Anda menyarankan untuk terus menyelesaikan-perayapan melalui host .. ( nslookup atau menggali mungkin ide yang lebih baik daripada ping). Bagi saya, ini tampak mubazir .. artinya, mengadakan polling skrip setiap X menit. Sepertinya hanya pencapaian kecil pada akhirnya .. Anda juga harus memutuskan, apakah Anda ingin menanyakannya NS server yang bertanggung jawab atas domain tersebut secara langsung atau melalui rantai DNS (non-otoritatif).

\n\n

Solusi yang lebih mudah mungkin saja mendapatkan layanan DNS yang lebih cepat. Saya tidak dapat mengingat nama situsnya, tetapi ada sesuatu yang melakukan penilaian latensi server DNS di wilayah Anda .. Agak akurat, dari yang saya ingat. Seseorang mungkin akan mengarahkan Anda ke sana.

\n\n

Meskipun, jika Anda menghadapi masalah konektivitas di lab - itu cerita lain. Bisakah Anda beralih ke server DNS alternatif? (Saya berasumsi Anda bisa, jika Anda memiliki akses ke / etc / hosts - kecuali lalu lintas entah bagaimana difilter untuk permintaan DNS).

Tidak. Ini duplikasi yang tidak berguna.

\n\n

Orang-orang terobsesi dengan file hosts, meskipun idenya telah digantikan pada awal 1980-an oleh DNS. Ini bahkan tidak seolah-olah orang-orang yang sama menjalankan sistem operasi yang miskin fitur seperti salah satu dari keluarga DOS atau DOS-Windows. Saat ini, bahkan pengguna Windows yang terobsesi dengan hosts menjalankan Windows NT, yang dilengkapi dengan klien DNS caching sebagai bagian standar dari sistem operasi. Ketika klien DNS caching diaktifkan - sebagaimana defaultnya - semua pencarian DNS oleh aplikasi di-cache secara lokal.

\n\n

Sistem operasi selain (edisi workstation) Windows NT dilengkapi dengan caching DNS server proxy (djbdns, BIND, MaraDNS, dan sejenisnya untuk Unices dan Linux, misalnya), yang tidak hanya menyimpan informasi secara lokal, tetapi sepenuhnya mampu melakukan keseluruhan Resolusi kueri DNS secara lokal juga. Jalankan server proxy DNS sebagai server proxy yang menyelesaikan, arahkan pustaka klien DNS ke sana, dan tidak perlu khawatir tentang kelemahan server proxy orang lain sama sekali .

\n\n

Dalam kedua kasus tersebut, jika Anda ingin menyimpan informasi DNS tertentu dalam cache siap untuk pengambilan segera setiap kali browser WWW mencoba pencarian DNS, cukup mencarinya terlebih dahulu dengan cara biasa . (Gunakan program yang menggunakan rutinitas pustaka klien DNS yang disediakan sistem. Program seperti sangat cacat nslookup memiliki pustaka klien DNS mereka sendiri yang terpisah, dan tidak akan berfungsi.) Ketahuilah bahwa beberapa organisasi sengaja menggunakan nilai TTL yang sangat pendek pada data DNS mereka. Mereka ingin Anda harus mencari hal baru setiap 30 detik (atau apa pun).
