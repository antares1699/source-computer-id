Saya telah menggunakan fungsi dasar SVN sebelumnya untuk memeriksa kode dan memeriksanya. Saya tidak pernah menyentuh cabang dan hal-hal yang lebih rumit seperti itu. Apakah mungkin untuk menggunakan SVN sehingga saya dapat menyimpan versi master dari file kode sumber, membuat versi yang sedikit berbeda untuk proyek yang berbeda, dan kemudian membuat perubahan apa pun pada versi master disalin ke versi lain tanpa menimpa perubahan dibuat untuk versi lain itu?

\n\n

Saya agak ragu ini mungkin untuk dengan mudah ditangani tanpa harus menangani konflik secara manual, jadi izinkan saya mengajukan pertanyaan alternatif: Jika saya membuat repositori dengan semacam pustaka kode umum dan kemudian membuat perubahan khusus proyek apa pun padanya di pisahkan file menggunakan warisan kelas, apakah mungkin untuk menyiapkannya sehingga ada dua repositori yang diperiksa di folder yang sama?

Ini adalah hal yang sulit dilakukan di SVN. Ini membutuhkan banyak penggabungan manual dan rentan terhadap kesalahan. Namun, jika Anda memiliki opsi, suka @Amenyebut , sistem kontrol versi terdistribusi menangani skenario ini dengan cukup baik. Secara khusus, saya tahu bahwa Git membuat ini sangat mudah.

\n\n

Itu git-rebase perintah melakukan apa yang Anda inginkan. Dengan asumsi Anda memiliki cabang utama atau "asal" dari sumber Anda, Anda cukup membuat cabang baru ("karya saya") untuk membuat perubahan khusus proyek Anda pada (diwakili oleh blok C6).

\n\n

\n\n

Sekarang, karena cabang utama kode terus berubah (revisi C3 & C4), Anda hanya perlu melakukan rebase cabang spesifik proyek Anda ke revisi baru dan Anda akan mendapatkan semua perubahan yang telah dibuat pada cabang utama di cabang proyek Anda. .

\n\n

\n\n

Masih ada kemungkinan harus melakukan penggabungan manual, tetapi kemungkinan untuk melakukannya jauh lebih rendah. Git menghilangkan banyak rasa sakit saat melakukan operasi penggabungan. Lihat Bab rebasing dari Git Book untuk info lebih lanjut.

Saya pikir ini cukup sulit dilakukan dengan subversi (setidaknya versi saat ini), jika bukan tidak mungkin, tetapi sistem kontrol versi terdistribusi akan menanganinya. Saya telah menggunakan Mercurial untuk menyelesaikan masalah ini:

\n\n

Di sisi "server":

\n\n
    \n
  • Buat repositori "master"
    • \n
  • Gandakan repositori itu untuk membuat versi repositori khusus proyek (penggandaan membuat duplikat dari seluruh repositori di tempat lain)
    • \n
\n\n

Di sisi "klien":

\n\n
    \n
  • Gandakan repositori proyek ke area pengembangan Anda dan periksa
    • \n
  • Lakukan perubahan, lakukan banyak hal secara lokal dan dorong kembali ke server sebagaimana mestinya
    • \n
\n\n

Jika terjadi perubahan pada master, mereka dapat "ditarik" ke dalam repositori khusus proyek, yang akan mengalir ke repositori lokal. Jika perubahan yang dibuat pada repositori khusus proyek dianggap cocok untuk "master", mereka dapat didorong kembali ke sana. Mercurial mengingat jalur antara repositori induk, jadi semuanya cocok satu sama lain. Penggabungan antara cabang (yang cenderung Anda gunakan lebih banyak) dan repositori umumnya bebas rasa sakit. Salah satu kelemahannya adalah Mercurial tidak mengizinkan update "cherry picking", tetapi itu bisa menjadi keuntungan karena itu membuat Anda terhindar dari potensi masalah dalam penggabungan di masa mendatang.

\n\n

Mungkin ada kurva pembelajaran yang lebih besar dalam melakukan semua ini, tetapi mungkin bermanfaat untuk diselidiki.
