Selingan singkat: %CD%

\n

Jika Anda tidak terlalu terbiasa dengan baris perintah Windows klasik (cmd.exe), variabel khusus %CD% akan memberi kita jalur absolut ke direktori kerja saat ini. Ingatlah itu untuk bagian berikut.\nJadi jika Anda saat ini berada di dalam LPT5, perintah echo %CD% akan menghasilkan keluaran C:\\test. Ini jalan pintas yang nyaman untuk percobaan keluar.

Seperti yang dapat kita peroleh dari artikel Project Zero, ada sejumlah cara untuk menghindari konversi nama jalur di tingkat subsistem Win32. Salah satu metode tersebut adalah prefiks \\\\?\\ yang secara internal langsung diterjemahkan menjadi 

echo NONSENSE > NUL\n
, yang pada versi Windows yang lebih baru identik dengan NUL. Ini disebut direktori objek (tolong jangan bingung dengan entitas sistem file , meskipun terminologi serupa!). Sekali lagi, WinObj, dan alat serupa memungkinkan Anda menyelidiki ruang nama pengelola objek.

Interlude: namespace dan terminal server "barang"

\n

Siapa pun yang telah melihat sejarah Windows NT, dan Windows 10 menelusuri akarnya kembali ke NT, akan mengingat saat ketika Anda harus secara terpisah melisensikan layanan terminal. Yaitu. kemampuan untuk terhubung ke satu mesin dari jarak jauh dengan pengguna yang berbeda.

Saya pikir itu adalah Windows XP yang akhirnya membawa ini untuk massa dengan cara memungkinkan untuk tetap masuk dengan beberapa akun pengguna secara bersamaan ("peralihan pengguna"), dan mungkin Windows 2000 atau 2003 Server memasukkan ini dalam edisi standar, meskipun CALs dibutuhkan di luar "kursi" minimum yang disertakan secara default.

\n

Di sinilah perbedaan antara \\?? dan \\GLOBAL?? berasal. \\GLOBAL?? adalah tampilan dari nama perangkat "DOS" yang digunakan bersama oleh semua sesi logon.

\n

\\?? saat ini terlihat di tautan simbolis (sekali lagi, bukan entitas sistem file! ) disebut \\DosDevices, yang memberikan petunjuk tentang asalnya. Di sinilah nama perangkat "DOS", seperti C: atau pemetaan drive jaringan berada. Pada sistem modern, C: pada gilirannya akan menjadi tautan simbolis ke \\Device\\HarddiskVolume1 (atau serupa). Biasanya itu adalah file sebenarnya objek perangkat yang dibuat oleh beberapa driver dalam tumpukan driver penyimpanan, dalam hal ini Sebaiknya menjadi driver sistem file NTFS.

Ohhhhh! Benar, warisan DOS dari Windows menyerang lagi. NTFS - kecuali secara aktif menonaktifkannya - memiliki kemampuan untuk membuat apa yang disebut 8.3 nama file pendek, sesuai dengan persyaratan nama file DOS.

\n

Dan begitulah cara kami dapat mengakses nama file yang tidak valid.

\n

Kesimpulan

\n

Sekarang ingat, pertanyaannya adalah tentang drive eksternal, yaitu lokal, NTFS. Ini berarti aturan yang baru saja kami amati untuk saham Samba mungkin tidak berlaku di sini.

\n

Bergantung pada driver yang digunakan untuk menyimpan file-file ini (yang dapat bervariasi menurut versi Linux / macOS, mis. Ntfs-3g atau driver pihak ketiga yang digunakan, mis. Driver Paragon) Saya melihat kemungkinan penyebab yang tersisa setelah melihat eksperimen di atas:

\n
    \n
  1. nama file berisi :, " atau \\\\?\\ ... ini sepertinya yang paling mungkin bagi saya, mengingat saya telah secara tidak sengaja menyalin dan menempelkan judul ebook sendiri, yang berisi karakter ini. Kita dapat mengesampingkan / dan karakter "terlarang" lainnya setidaknya lebih kecil kemungkinannya.
    2. \n
  2. sisi Windows dan macOS melihat nama yang tidak valid, mencoba melihat nama DOS 8.3, tetapi tidak ada yang dibuat. Ini agak tergantung pada versi Windows yang tepat dan konfigurasinya dan karena saya tidak memiliki perangkat macOS, saya juga tidak dapat menguji skenario itu. Juga, saya tidak yakin apakah pada sistem Windows di mana 8,3 nama diaktifkan Windows secara retroaktif akan pergi dan menghasilkan nama 8.3 jika, katakanlah, pihak Linux melewatkan bagian itu. Karena jika saya ingat dengan benar pengandar NTFS memutuskan apakah catatan masing-masing ("atribut") akan diisi .
    3. \n
  3. panjang nama jalur telah terlampaui. Saya pikir melebihi panjang segmen jalur adalah suatu kemustahilan, karena NTFS tidak memungkinkan Anda menyimpan lebih dari 255 nilai 16-bit untuk segmen jalur, tetapi keseluruhan panjang jalur mungkin juga telah terlampaui (lihat Link ini ).
    4. \n
\n

Untuk skenario pertama, saya akan merekomendasikan penggunaan fslint.dll di sisi Linux untuk membersihkan nama file (dan folder). Ada alat serupa lainnya dan YMMV, pilihlah.

\n
