Selama beberapa tahun terakhir, saya telah mengumpulkan sekumpulan hard disk yang cukup heterogen dengan berbagai ukuran dan kecepatan untuk menyimpan data pribadi saya. Saya berencana untuk menempatkannya di filer Linux yang dibuat sendiri untuk mengurangi kerumitan mereplikasi data lokal secara manual, mengurangi file duplikat, dan memanfaatkan sumber daya yang diberikan dengan lebih baik. Saya juga berharap bahwa di tahun-tahun mendatang kebutuhan penyimpanan saya akan meningkat. Oleh karena itu, dimungkinkan untuk menambahkan disk secara dinamis dan juga menghapus drive tunggal untuk menggantinya dengan yang lebih baru dan lebih besar.

\n\n

Jika saya tidak salah, opsi paling umum untuk membuat filer adalah menggunakan RAID (perangkat lunak) untuk meningkatkan keandalan dan skema pencadangan tambahan pada drive eksternal atau yang dapat dilepas untuk mencegah hilangnya data penting secara tidak sengaja.

\n\n

Karena RAID5 / RAID6 membutuhkan drive dengan ukuran yang sama, hal itu tidak memenuhi kebutuhan saya akan skema penambahan / penghapusan disk yang lebih dinamis. Jadi saya mencari sistem file FLOSS atau lapisan abstraksi perangkat blok yang menyediakan:

\n\n
    \n
  • penambahan dan pemindahan hard disk secara dinamis dimungkinkan
    • \n
  • replikasi / redundansi mirip dengan RAID5, mungkin dapat disesuaikan per file atau direktori
    • \n
  • tidak diperlukan mesin tambahan (khusus khusus) (tetapi kemungkinan untuk menambahkan mesin kedua nanti jika diperlukan akan menyenangkan tetapi tidak perlu)
    • \n
\n\n

Sederhananya, percaya atau tidak, solusi perangkat keras adalah lebih mudah untuk dibangun kembali dan lebih terpercaya daripada solusi perangkat lunak, dan sering kali solusi yang ada lebih murah untuk diterapkan .

\n\n

Membangun sistem penyimpanan Anda umumnya bukanlah ide yang baik kecuali Anda sangat ahli dalam seni manajemen penyimpanan. Secara umum, berlaku diktum lama "Jika Anda harus memintanya, itu bukan untuk Anda".

\n\n

Intinya: Saya akan merekomendasikan membeli harddisk baru (Akhir-akhir ini harganya lebih murah dibandingkan dengan data saya - jika data Anda tidak layak untuk membeli satu set harddisk akhir-akhir ini, Houston kami punya masalah .) \n\n

Lebih penting lagi, perhatikan bahwa pencadangan dan redundansi adalah dua entitas terpisah dalam manajemen penyimpanan. Redundansi adalah untuk mempertahankan layanan ONLINE jika salah satu peralatan mati, dan cadangan jika semuanya gagal. Misalnya, jika catu daya tiba-tiba gagal dan mencoba memompa beberapa 100V AC ke rel 12V (Bukan itu mungkin, tapi ...) tidak peduli RAID5, tidak peduli RAID6 - semua akan hilang. Saya akan melihat ke ZFS. Ini (atau beberapa implementasinya) menawarkan beberapa hal yang Anda minta secara khusus (redundansi di media heterogen, kemampuan untuk dengan mudah mengatasi hilangnya spindel (jika ada data yang berlebihan)).

Seharusnya mungkin (dan saya minta maaf, saya tidak terlalu jauh ke dalam LINUX) untuk membangun beberapa volume RAID di sekumpulan disk drive yang heterogen; sehingga memaksimalkan penggunaan ruang sekaligus menjaga redundansi.

\n\n

Bayangkan Anda memiliki drive 3x1TB, 2X2TB, dan 1 drive 4TB.\nAnda dapat membangun RAID 5 volume ruang 5TB menggunakan 1TB dari semua 6 drive.\nAnda dapat membuat RAID 5 lain dengan volume 2TB menggunakan 1TB dari masing-masing 3 drive yang tersisa.

\n\n

Di server bisnis kecil Windows, volume ini kemudian direntang; muncul sebagai satu volume. Saya berharap saya bisa lebih membantu dalam membuat rekomendasi khusus tentang bagaimana melakukannya di LINUX.

\n\n

Jika Anda menambahkan, katakanlah, drive 4TB lainnya: Anda kemudian dapat mengembangkan kedua larik sebesar 1TB dan menambahkan larik 2TB ketiga menggunakan ruang 2TB yang tersedia di setiap drive 4TB.

\n\n

Mengenai perangkat keras-vs-perangkat lunak: Perangkat keras tidak diragukan lagi lebih cepat; tetapi harganya lebih mahal dan, jika pengontrol perangkat keras itu sendiri gagal, mungkin mengharuskan Anda menemukan pengontrol yang identik atau kehilangan semua data.

\n\n

Seperti biasa: redundansi bukanlah pengganti cadangan.
