Jika keamanan bukan persyaratan, seperti dalam kasus Anda menjalankan OS tervirtualisasi. Anda dapat membuka jendela X11 langsung ke layar Ubuntu non-VM Anda. Ini seharusnya secara teori memberi Anda kecepatan terbaik dan latensi terendah, meskipun dalam beberapa kasus saya telah memperhatikan bahwa VNC melakukan pekerjaan yang lebih baik. Satu-satunya kelemahan adalah bahwa tanpa beberapa aplikasi pembantu tambahan, Anda tidak dapat menutup koneksi aplikasi dan menghubungkan kembali nanti.

\n\n

Mari kita asumsikan bahwa VM Ubuntu Anda memiliki alamat IP 10.0.0.2 dan Ubuntu "asli" memiliki antarmuka virtual IP 10.0.0.1 dan merupakan GW untuk mesin virtual (ini akan menjadi sesuatu yang Anda miliki dengan mode NAT. Dengan mode jembatan , kedua OS tersebut memiliki IP yang diberikan beberapa server DHCP eksternal, atau keduanya diatur secara manual).

\n\n

Di Ubuntu non-VM Anda, Anda perlu menggunakan perintah xhost untuk mengizinkan koneksi X11 yang masuk dari VM, untuk melakukan penulisan ini pada aplikasi terminal: 

xhost +10.0.0.2\n

Kemudian hubungkan ke mesin virtual dengan ssh atau dengan terminal mesin virtual dan buka aplikasi terminal dan tulis: 

export DISPLAY=10.0.0.1:0.0\n

kemudian ketika Anda menjalankan aplikasi dari jendela terminal itu, itu membuka jendelanya langsung di Ubuntu non-VM Anda.

Anda dapat meminta kompresi ssh menggunakan -C:

\n\n

ssh -X -C myhost

\n\n
\n

-C Meminta kompresi semua data (termasuk stdin, stdout, stderr, dan data untuk koneksi X11 dan TCP yang diteruskan). Algoritma kompresi sama dengan yang digunakan oleh gzip (1), dan "level" dapat dikontrol oleh\n Opsi CompressionLevel untuk protokol versi 1. Kompresi diinginkan pada jalur modem dan koneksi lambat lainnya, tetapi hanya akan memperlambat sesuatu pada jaringan yang cepat. Nilai default dapat diatur berdasarkan host-by-host\n di file konfigurasi; lihat opsi Kompresi.

\n
\n\n

Atau Anda mungkin ingin mencoba FreeNX, yang lebih cepat dari VNC
