Alasannya adalah bahwa Recycle Bin adalah folder "khusus" yang fungsinya hanya berfungsi di komputer lokal saat memproses perintah hapus. Perbedaan antara Recycle Bin dan alat undelete seperti Recuva adalah Recycle Bin adalah tempat yang "aman" untuk menyimpan file yang mungkin tidak ingin Anda hapus. File di dalamnya dijamin dapat dipulihkan sampai Anda sengaja mengosongkannya. Ketika sebuah file "sebenarnya" dihapus, itu mungkin dapat dipulihkan dengan alat seperti Recuva, tetapi tidak ada jaminan, dan dengan menggunakan itu berarti ada sesuatu yang tidak beres.

\n\n

Intinya adalah, Recycle Bin adalah fitur kenyamanan yang membantu Anda membatalkan penghapusan file Anda sendiri di komputer Anda sendiri. Itu tidak berfungsi di komputer lain melalui jaringan.

\n\n

Jika Anda tertarik dengan detailnya, baca terus ... Jika tidak, terima saja ini sebagai jawabannya.

\n\n
\n\n

TL; DR

\n\n

Ada direktori biasa (tersembunyi) di root setiap volume di komputer Anda yang disebut $ Recycle.Bin. Di dalam direktori ini terdapat subdirektori yang mewakili SID (pengenal keamanan) dari setiap pengguna di sistem yang pernah menghapus file (setiap pengguna memiliki SID unik). Windows merepresentasikan Recycle Bin di desktop Anda sebagai komposit "yang diproyeksikan" dari semua subdirektori SID pengguna Anda dari setiap direktori $ Recycle.Bin pada semua drive di komputer Anda.

\n\n

Misalnya, SID pengguna Anda adalah S-1-5-21-123456789-123456789-123456789-12345, dan komputer Anda memiliki 3 volume lokal (C:, D:, dan E:), yang dapat berupa 3 hard drive berbeda, 3 partisi pada drive yang sama, atau kombinasi keduanya. Di setiap hard drive, Anda akan memiliki yang berikut:

\n\n

C:\\$Recycle.Bin\\S-1-5-21-123456789-123456789-123456789-12345
D:\\$Recycle.Bin\\S-1-5-21-123456789-123456789-123456789-12345
E:\\$Recycle.Bin\\S-1-5-21-123456789-123456789-123456789-12345

\n\n

Windows akan menunjukkan kepada Anda satu ikon Recycle Bin di desktop Anda yang berisi file dari ketiga direktori tersebut. Ada beberapa hal yang harus Anda waspadai dengan penyiapan ini:

\n\n
    \n
  • Recycle Bin di desktop Anda hanya berisi file yang ANDA hapus. Jika ada akun pengguna lain di komputer, Anda tidak akan melihat file MEREKA dihapus, mereka juga tidak dapat melihat milik Anda. Ini berlaku bahkan jika Anda menghapus file dari direktori yang sebelumnya dapat diakses oleh semua pengguna. Ini memiliki efek samping memberikan satu pengguna kemampuan untuk "menangkap" file; tidak ada yang bisa memulihkan file itu kecuali orang yang awalnya menghapusnya.
    • \n
  • Saat Anda "menghapus" file, Windows sebenarnya melakukan operasi "pemindahan" ke direktori $ Recycle.Bin Anda, mengganti namanya menjadi sesuatu yang unik, dan menambahkan sedikit metadata yang menunjukkan nama aslinya dan dari mana asalnya (jadi Anda dapat memulihkannya nanti). Inilah sebabnya mengapa Anda tampaknya memiliki banyak file dengan nama yang sama di direktori yang sama, yang biasanya tidak dapat Anda lakukan.
    • \n
  • Operasi "pemindahan" dari satu lokasi ke lokasi lain pada volume yang SAMA bersifat instan karena hanya penunjuk sistem file yang dimodifikasi sehingga membuat file tampak seolah-olah berada di direktori yang berbeda. Sistem tidak harus benar-benar memindahkan konten file ke lokasi baru. Namun, Anda tidak dapat "memindahkan" file dari satu drive ke drive lain (Windows melakukan penyalinan diikuti dengan penghapusan yang asli dalam hal ini). Inilah sebabnya mengapa setiap drive memiliki direktori $ Recycle.Bin sendiri. Anda tidak ingin operasi "hapus" menyalin file yang berpotensi besar ke drive lain, yang akan memakan waktu dan drive lain mungkin tidak memiliki cukup ruang.
    • \n
\n\n

Selain itu;

\n\n
    \n
  • Protokol SMB (yang ada di belakang berbagi file Windows) tidak ditemukan oleh Microsoft, dan dirancang untuk menjadi komputer-agnostik pada waktu sebelum Windows menjadi sesuatu. Dengan demikian, ia sengaja tidak memiliki konsep sistem file yang mendasari komputer tempat share tersebut di-host. Inilah sebabnya mengapa sistem non-Windows dapat menjadi host dan terhubung ke share SMB; protokol memisahkan teknis dari media penyimpanan yang sebenarnya, memungkinkan perangkat yang tidak kompatibel (bahkan telepon dan printer) untuk berbagi file satu sama lain.
    • \n
  • Ini juga mengapa Anda tidak dapat "memindahkan" file di antara dua berbagi file yang berbeda. Apa yang sebenarnya Anda lakukan adalah mengunduh file ke komputer Anda sendiri, mengunggah file kembali ke komputer jarak jauh di lokasi baru, dan kemudian menghapus file aslinya. Ini terjadi meskipun dua bagian yang berbeda sebenarnya hanya nama yang berbeda untuk folder fisik yang sama.
    • \n
  • Direktori $ Recycle.Bin tidak dapat dibagikan, karena isinya hanya berarti bagi komputer tempat sebenarnya disimpan.
    • \n
\n\n

Saat Anda menghapus file melalui jaringan, tugas komputer Anda adalah "memarkir" file di Recycle Bin, bukan di komputer jarak jauh. Ini tidak mungkin bekerja melalui jaringan. Inilah alasannya:

\n\n
    \n
  • Komputer Anda tidak memiliki akses ke Recycle Bin komputer jarak jauh, karena tidak dapat dibagikan.
    • \n
  • Meskipun itu dapat dibagikan, Anda mungkin tidak memiliki akun di komputer itu untuk menyimpan file (seperti jika pembagian mengizinkan "publik", pengguna anonim untuk mengaksesnya misalnya).
    • \n
  • Bahkan jika Anda memang memiliki akun, berapa volume Recycle Bin yang digunakan? Ingat, "memindahkan" file antara berbagi SMB sebenarnya adalah salinan dua arah. Komputer Anda tidak tahu tentang penyimpanan yang mendasari komputer jarak jauh.
    • \n
\n\n

Beberapa di antaranya dapat dipecahkan jika protokol SMB menyertakan beberapa fungsionalitas untuk komputer jarak jauh untuk menangani operasi penghapusan dengan caranya sendiri. Tapi itu punya masalah sendiri:

\n\n
    \n
  • Bagaimana komputer lain yang Anda gunakan tahu untuk melihat Recycle Bin komputer lain untuk file yang mungkin telah Anda hapus darinya?
    • \n
  • Bagaimana Anda mengosongkan Recycle Bin jika Anda atau mesin jarak jauh sedang offline pada saat itu? Bagaimana jika tidak lagi memiliki akses ke mesin jarak jauh itu? Anda tidak akan dapat mengosongkan Recycle Bin Anda secara permanen.
    • \n
  • Jika Anda memulihkan file yang dihapus, kemana perginya? Nama yang dibagikan, nama server, dan alamat IP dapat berubah setiap saat, dan bagian yang berbeda pada mesin yang berbeda dapat mengarah ke tempat fisik yang sama.
    • \n
  • Bagaimana jika remote share tidak ada di komputer Windows (atau bahkan komputer sama sekali)? Ini tidak akan memiliki konsep Recycle Bin untuk menempatkan file ke dalamnya.
    • \n
  • Banyak masalah lain yang bahkan tidak saya pikirkan.
    • \n
\n\n

Banyak dari ini sulit untuk dipahami ketika Anda hanya memiliki dua komputer dan Anda satu-satunya orang yang menggunakannya. Tapi semua ini menjadi masalah besar dalam skema yang lebih luas.

\n\n

Inilah sebabnya, hampir 40 tahun (!) Setelah protokol dikembangkan, di sana masih bukanlah cara yang diterima secara universal untuk memulihkan file yang terhapus melalui jaringan.
