debug1: Local version string SSH-2.0-OpenSSH_6.2\nssh_exchange_identification: read: Operation timed out\n

Berdasarkan pelacakan debug Anda, Anda mengelola untuk menyambung ke server SSH jarak jauh, dan server tidak memutuskan sambungan, tetapi server tidak mengirimkan string identifikasi perangkat lunaknya. Ini adalah hal pertama yang dilakukan oleh klien dan server, dan ini dilakukan dalam teks yang jelas.

\n\n

Saya pikir masalahnya akan turun ke salah satu dari tiga hal:

\n\n
    \n

  1. Beberapa perangkat jaringan antara Anda dan server mengganggu koneksi. Misalnya, jika host jarak jauh berada di belakang router NAT, penerusan port untuk port 22 mungkin salah dan Anda menyambungkan ke layanan yang salah.

    2. \n

  2. Program server SSH pada host jarak jauh mungkin macet atau tidak berfungsi. Saya telah melihat hal semacam ini terjadi misalnya ketika host sangat kelebihan beban atau kehabisan memori virtual. Atau server mungkin menggunakan sesuatu seperti Wrapper TCP , dan melakukan kueri DNS pada IP klien Anda yang membutuhkan waktu lama untuk diselesaikan.

    3. \n

  3. Host jarak jauh disiapkan dengan cara yang tidak biasa, dan layanan yang berjalan pada port 22 bukanlah server SSH.

    4. \n
\n\n

Jika Anda dapat mengakses server, fokuskan pemecahan masalah Anda di sana. Temukan log untuk server ssh - seharusnya ada di /var/log dan lihat apakah log telah mencatat apa pun tentang upaya koneksi yang gagal ini. Coba jalankan ssh ke localhost dari server dan lihat apakah itu berfungsi dengan baik.

\n\n

Jika Anda memiliki akses root ke server, Anda dapat mencoba untuk memulai contoh debugging sshd untuk melihat apa yang dicatatnya ketika klien Anda terhubung. Hentikan server normal sshd, dan di jendela root terminal, jalankan /path/to/sshd -d. Ini akan menjalankan salinan sshd yang akan menerima satu koneksi dan mencetak informasi debugging ke jendela terminal. Lihat apakah Anda dapat mereproduksi masalah, dan kemudian periksa apa yang dicatat oleh server.

\n\n

Jika Anda tidak dapat membuat server ssh normal offline, Anda dapat menjalankan sshd pada port lain: /path/to/sshd -p 42 -d menjalankan salinan sshd listening pada port 42. Tentukan nomor port yang sama saat menjalankan klien ssh: ssh -p 42 user@host. Jika masalah Anda disebabkan oleh perangkat jaringan yang mengganggu, ini mungkin berperilaku berbeda dari koneksi ke port 22.
