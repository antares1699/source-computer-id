Beberapa pengguna kami di organisasi kami telah mulai melaporkan kesalahan aneh saat memasang produk kami (detail produk disunting dari tangkapan layar):

\n\n

\n\n

Pesan kesalahan asli dalam bahasa Inggris adalah "Direktori instalasi harus di drive lokal".

\n\n

Mengubah jalur ke sesuatu tanpa spasi, folder root alternatif atau bahkan drive fisik lainnya pada mesin tidak mengubah apa pun. Penginstal masih bersikeras bahwa jalur tersebut bukanlah jalur lokal.

\n\n

Penginstal ini dibuat dengan WIX, dan langkah spesifik dalam proses dibuat dengan tindakan kustom WixUIValidatePath. Bug sebelumnya telah dikirimkan , itu juga menunjukkan bahwa kesalahan terkait dengan pembatasan pada penginstal ( terima kasih kepada peterflynn di GitHub karena telah mengarahkan saya ke arah itu ).

\n\n

Masalah hanya terjadi pada beberapa komputer, dan tampaknya tidak ada yang memiliki masalah di Windows 10. Saya belum dapat menguji apakah itu terjadi di versi Windows lain bagian dari 8.1, karena itu dan Windows 10 adalah satu-satunya versi kami. Lari.

Setelah banyak debugging, kami mempersempitnya menjadi masalah dengan hak atas folder temp dengan pengguna saat ini. Langkah yang berbeda menyelesaikan masalah di komputer yang berbeda. Di sini saya akan mencantumkan perbaikan dari pendekatan yang paling tidak invasif hingga yang paling invasif:

\n\n
    \n
  1. Pastikan penginstal dijalankan sebagai administrator
    Ini adalah sesuatu yang sudah kami butuhkan, tetapi beberapa pengguna cenderung melupakannya. Ini juga merupakan solusi paling umum untuk masalah yang saya temukan saat mencari solusi.
    Cara termudah untuk memastikan ini, adalah dengan menekan WinKey dan ketik cmd untuk menemukan prompt perintah, lalu tekan CTRL + SHIFT + ENTER untuk membuka prompt perintah sebagai administrator, lalu navigasikan ke folder msi ( untuk petunjuk navigasi, lihat tautan ini ) dan menjalankannya dari sana.
    2. \n
  2. Instal KB3072630 secara manual
    Beberapa pengguna kami, karena alasan tertentu, tidak diterima dengan benar pembaruan khusus ini dari MS yang sepertinya menyelesaikan masalah! Di sinilah sebagian besar pengguna menyelesaikan masalah mereka.
    3. \n
  3. Jalankan penginstal secara diam-diam
    Bug terkait dengan komponen UI WIX, jadi menjalankan MSI secara diam-diam tidak akan mengaktifkan komponen UI, dan oleh karena itu bug juga tidak akan terpicu.\nAnda dapat melakukan ini melalui commandprompt seperti yang ditentukan pada langkah di atas, tetapi dengan parameter yang ditambahkan ini ke MSI:
    /qn+
    \n Terima kasih kepada Kr3m di GitHub untuk pendekatan ini
    4. \n
  4. Jalankan penginstal dengan pemeriksaan UI dinonaktifkan
    Komponen WIX UI yang mengganggu semuanya juga dapat dinonaktifkan. PERINGATAN: Ini juga akan menonaktifkan pemeriksaan kewarasan pada jalur instalasi, dan mungkin mengacaukan proses instalasi! Ini dapat dilakukan dengan parameter baris perintah lain:
    WIXUI_DONTVALIDATEPATH=\"1\"
    \n Terima kasih kepada peterflynn di GitHub untuk pendekatan ini
    5. \n
  5. Nonaktifkan LUA
    Satu-satunya cara yang pasti untuk menghilangkan masalah tersebut, adalah dengan menonaktifkan LUA sepenuhnya, yang merupakan standar untuk beberapa lingkungan kami, agar dapat bekerja dengan fitur lama. Menonaktifkan LUA sepenuhnya memerlukan perubahan manual pada registri, jadi pastikan Anda tahu apa yang Anda lakukan, dan konsekuensinya juga. Instruksi dapat ditemukan di sini di forum MS .
    6. \n
\n\n

Satu hal yang tidak kami uji, yang mungkin juga bisa menyelesaikannya, adalah membuat ulang pengguna di komputer, karena menurut kami ini ada hubungannya dengan hak akses ke folder temp milik pengguna itu sendiri. Mereka mungkin dibuat ulang dengan benar melalui rekreasi pengguna.
