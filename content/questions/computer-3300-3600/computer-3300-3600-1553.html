Tidak ada solusi dengan klien NFS Microsoft di Windows 7

\n\n

Tidak ada cara untuk memecahkan masalah Anda di bawah pengaturan ini, bahkan metode yang diberikan dalam "Jawaban Benar" dari pertanyaan ini. Ini bukan kesalahan, ini dirancang untuk menciptakan penghalang yang Anda hadapi. Tidak mungkin perancang NFS Client menawarkan opsi untuk menggunakan BIG5, EUC-KR, EUC-JP, ksc5601, GB18030, SHIFT-JIS dan lupa UTF-8 dan terus lupa menambalnya dalam satu dekade . Semua pengkodean ini digunakan oleh berbagai versi Windows kecuali UTF-8, bukan kebetulan. Faktanya, Microsoft bermaksud untuk mendapatkan UNIX dan Windows lama, untuk mendorong migrasi dari Unix ke Windows, dan harus diukur dengan hati-hati bahwa hal ini tidak memberikan lingkungan yang mudah bagi kompetitornya, Linux. Fitur apa pun yang membuat Linux berbeda dari Unix atau Unix modern berbeda dari yang lama-unix-to-be-replace dikecualikan. Ini presisi dalam pemasaran, pekerjaan selesai dengan baik.

\n\n

Izinkan saya menunjukkan kepada Anda bagaimana setiap cara diblokir:

\n\n

Untuk mengatasinya, sisi klien tidak boleh dilakukan:

\n\n

Idealnya adalah menemukan beberapa perangkat lunak yang dapat secara dinamis mentranskode nama file, misalnya dengan menyisipkan sendiri lapisan antara FS dan OS. Atau, temukan beberapa perangkat lunak yang dapat menyiapkan lingkungan untuk perangkat lunak lain untuk menggunakan sistem file itu. Tidak ada perangkat lunak Windows yang dapat melakukan ini. Ini tidak berhasil, karena tidak seperti Linux yang memungkinkan setiap aplikasi menggunakan pengkodean yang berbeda, Windows mengkonfigurasi pengkodean untuk FILESYSTEM.

\n\n

Untuk mengatasinya, server tidak boleh digunakan:

\n\n

Kebanyakan server NFS adalah Linux. Secara teori, seseorang dapat menggunakan fuse-convmvfs untuk membuat mirror dari filesystem dan mengekspornya. Oleg Lobach mendemonstrasikan kode untuk melakukan ini, dan itu ditandai benar tanpa tes. Itu tidak berfungsi karena server NFS Linux hanya dapat mengekspor sistem file FUSE di NFSv4, bukan di NFSv2 / v3, sementara itu, Windows 7 (serta Layanan untuk Unix v3.5) hanya mendukung NFSv2 / v3, bukan NFSv4. Saya mendemonstrasikan batasan ini secara rinci di jawaban lain.

\n\n

Masalahnya dijabarkan dalam TechNet :

\n\n
\n

Nama file POSIX bukanlah ASCII tetapi aliran byte . Hal yang sama berlaku untuk\n nama file pada sistem file NFS. Interpretasi aliran byte sebagai\n nama file dalam kumpulan kode tertentu ditinggalkan sebagai latihan untuk klien\n aplikasi. Jika aliran byte nama file mewakili string UTF-8 yang valid,\n tetapi aplikasi klien telah menyetel kumpulan kode ke, katakanlah, ISO-8859-5, apa saja\n Masalahnya adalah kesalahan aplikasi.\n Tapi kita sedang berbicara tentang Windows. Di Windows, interpretasi file\n nama file tidak diserahkan ke aplikasi. Sebaliknya, aliran byte nama file\n dikirim dari server NFS ke mesin klien Windows harus dikonversi\n ke string UTF-16 agar dapat dicerna oleh OS. Oleh karena itu interpretasinya\n aliran byte nama file harus dilakukan oleh layanan klien NFS,\n dengan basis per titik pemasangan. Aplikasi tidak memiliki suara di dalamnya.\n Dan inilah masalahnya. Jika kode remote yang digunakan untuk membuat nama file adalah\n UTF-8, seperti default selama bertahun-tahun di dunia POSIX, tidak ada\n kesempatan untuk mendapatkan nama file yang benar dari sudut pandang aplikasi Windows,\n karena interpretasi nama file sudah dilakukan oleh Windows\n Klien NFS yang tidak mengizinkan konversi dari UTF-8 ke UTF-16. Di\n Faktanya, klien NFS hanya mendukung kumpulan kode yang dibatasi\n konversi ke UTF-16, semuanya agak kuno.\n Ini bukan hanya skenario tertentu. Anda memiliki masalah ini secara keseluruhan\n skenario di mana Anda mengakses share NFS pada mesin yang disetel untuk menggunakan MBCS\n seperti UTF-8 atau GB-18030 dari klien Windows NFS.

\n\n

Ada solusi jika Anda memilih untuk beralih:

\n\n

Solusi terbaik tentu saja meninggalkan Windows. Ini tidak penting saat ini, dan tren global tidak lagi merilis perangkat lunak khusus Windows. Jika Anda terjebak dengan aplikasi Windows lama, Anda dapat mengubah kebutuhan Anda:

\n\n
    \n

  • Ganti NFSv2 / v3 Client: Anda dapat menginstal dokan user-land mount driver untuk Windows, kemudian Neko NFS drive, yang memetakan NFS ke device driver dengan opsi "Unicode". Klik opsi itu agar UTF8 berfungsi. Ini mendukung NFSv2 dan v3, tetapi tidak mendukung v4. Itu belum mendukung tautan simbolis (seperti yang dilakukan klien Microsoft).

    • \n

  • Siwtch NFSv4 Client: Anda dapat meninggalkan NFS Client bawaan untuk implementasi referensi gratis dari NFS 4.1 untuk Windows oleh UMICH CITI , meskipun sulit dipasang, sebaiknya hilangkan masalah (karena NFS4 secara eksplisit memerlukan UTF-8) atau memungkinkan untuk menggunakan solusi fuse-convmv yang disebutkan di atas.

    • \n

  • Switch OS: Windows Server 2012 menambahkan dukungan untuk server NFS 4.1, jadi ada kemungkinan mereka menyertakan klien NFS 4.1 juga.

    • \n
\n\n

Kedua sakelar tidak akan berfungsi jika server Anda hanya mendukung NFSv3, misalnya, jika Anda menggunakan OpenWRT (bahkan versi terbaru server OpenWRT masih tidak dapat melakukan NFSv4).
