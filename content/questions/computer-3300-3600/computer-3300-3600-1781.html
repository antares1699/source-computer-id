Meskipun server DNS Google 8.8.8.8 adalah server DNS alternatif yang populer daripada yang disediakan oleh ISP Anda, implikasi privasinya terlalu mengganggu bagi saya. Saat mencari privasi yang menghormati server DNS alternatif, saya menemukan bahwa kumpulan OpenNIC cukup populer.

\n\n

Saya tidak yakin bagaimana perasaan saya tentang kepercayaan dari server ini. Mereka telah menerapkan ekstensi mereka sendiri ke zona akar di luar otoritas IANA. Apa yang akan menghentikan mereka (selain pengawasan publik) dari mendefinisikan ulang bagian dari zona akar yang didefinisikan oleh IANA?

\n\n

Saya memahami bahwa meneruskan kueri DNS langsung ke server nama root DNS dalam skala besar dianggap berbahaya, tetapi tampaknya satu-satunya cara untuk memastikan hasil yang benar * dan pribadi adalah dengan memiliki pemecah cache dan rekursif di jaringan lokal Anda. Untuk satu jaringan rumah, ini tidak akan membebani server root, tetapi pasti tidak akan menskalakan ke setiap rumah tangga.

\n\n

Jadi, haruskah saya menggunakan OpenNIC atau meneruskan ke server root? ISP saya sendiri tidak mungkin karena mereka secara rutin membajak DNS.

\n\n

* tanpa adopsi DNSSEC yang meluas, yang akan menyelesaikan aspek kebenaran tetapi bukan aspek privasi.

Pertanyaan tentang kebenaran bukanlah masalah teknis pada intinya, jadi Anda tidak akan pernah bisa sepenuhnya menjawab pertanyaan seperti "Haruskah saya mempercayai X", khususnya jika Anda menambahkan "... untuk tidak melakukan tindakan Y di masa depan yang jauh".

\n\n

Terutama karena dalam pertanyaan Anda, Anda tampak tidak yakin tentang penyedia itu sendiri, dan apa yang terjadi di jalur antara penyedia dan Anda.

\n\n

Jika Anda ingin lebih mengontrol proses resolusi Anda, Anda tidak memiliki pilihan lain selain menjalankan server nama caching rekursif Anda sendiri, baik di host Anda secara langsung, atau di server lain yang Anda percayai. Terutama jika Anda ingin memastikan sepenuhnya untuk menggunakan fitur yang disediakan oleh DNSSEC: jika Anda menggunakan server nama yang memvalidasi jauh, Anda mempercayainya untuk melakukan semua penghitungan DNSSEC dengan benar untuk Anda.

\n\n

Jadi saya bahkan tidak akan mencoba menilai apakah 1.1.1.1 (CloudFlare) atau 8.8.8.8 (Google) atau 9.9.9.9 (IBM + PCH + GlobalCyberAlliance) atau OpenNIC atau lainnya di https://en.wikipedia.org/wiki/Public_recursive_name_server atau di tempat lain dapat dipercaya atau lebih dapat dipercaya daripada yang lain. Ini juga merupakan pendapat yang sangat pribadi (kepada siapa Anda memberikan kepercayaan Anda), dan itu berubah seiring waktu.

\n\n

Penegasan Anda "tetapi itu pasti tidak akan berskala untuk setiap rumah tangga." tidak begitu jelas. Pergerakan semakin banyak bagi orang-orang untuk menangani resolusi DNS mereka di rumah (atau meneruskan ke salah satu resolusi publik sebelumnya), dan server root memiliki banyak kapasitas. Perhatikan bahwa masalahnya mungkin tidak terletak di sana pada kenyataannya karena file zona ini bergerak lambat, kecil, dan di-cache di mana-mana. Masalahnya bisa jauh lebih banyak di beberapa server nama TLD, seperti .COM, di mana file zona memiliki jutaan entri dan perubahan reguler yang mungkin tidak kecil.

\n\n

Anda memiliki berbagai opsi di atas meja, yang terkadang dapat Anda padu padankan:

\n\n
    \n
  1. Menggunakan Minimalisasi QNAME (didukung oleh beberapa layanan publik di atas) pada server nama yang Anda gunakan. Ini memberikan lebih sedikit informasi untuk setiap server nama sementara membiarkan protokol DNS berfungsi persis seperti sebelumnya
    2. \n
  2. Anda dapat menggunakan standar sekarang DNS melalui TLS untuk dapat menanyakan semua nameserver yang menawarkan itu (sekali lagi beberapa server publik melakukannya, atau berencana melakukannya) atau bahkan DNS "segera" melalui HTTPS. Dengan melakukan itu, tentu saja, Anda hanya memindahkan masalah: Anda aman dari pembajak di jalur, tetapi Anda perlu menetapkan otentikasi titik akhir yang Anda tukar; sekali lagi, lebih mudah jika Anda mengelolanya sendiri.
    3. \n
  3. Beberapa menyarankan untuk hanya menggunakan "beberapa" server DNS publik, secara acak (sehingga tidak satupun dari mereka mendapatkan semua lalu lintas Anda), dan bahkan membandingkan hasil
    4. \n
  4. Anda juga memiliki beberapa alat yang lebih halus, seperti Pendek dan gemuk (menggunakan getdns API) yang mencoba memberi Anda fitur terbaik dari segi privasi tetapi juga dapat dikonfigurasi untuk kembali ke mekanisme sebelumnya yang tidak aman jika Anda lebih mengutamakan ketersediaan daripada keamanan. Perangkat lunak seperti dnssec-trigger juga mencoba memberi Anda manfaat DNSSEC terlebih dahulu dengan menggunakan server nama default Anda dan memeriksa apakah memang berfungsi dengan benar dan akan menangani permintaan itu sendiri jika diperlukan.
    5. \n
  5. Untuk menjadi lengkap, saya perlu membuat daftar DNSCrypt (terbuka tapi tidak standar) yang bertujuan untuk mencegah spoofing. Namun Anda memerlukan klien dan server khusus untuk berkomunikasi menggunakan protokol ini.
    6. \n
\n\n
