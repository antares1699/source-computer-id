Penggunaan file "sementara" kembali ke masa ketika komputer menjalankan MS-DOS dan memiliki memori fisik 640 kB dan tidak ada paging. Memainkan tetris memori dengan driver perangkat dan TSR, dengan putus asa mencoba membebaskan beberapa RAM, adalah hal yang biasa (meskipun bukan "favorit") di masa lalu bagi pemilik PC.

\n\n

Saat ini,% TEMP% masih berguna sebagai tempat menyimpan file yang tidak perlu disimpan secara permanen. Misalnya, saat Anda mengunduh dokumen dari situs web dan memilih "Buka" alih-alih "Simpan", dokumen itu masih harus disimpan ke file secara lokal sebelum diberikan ke program - ini adalah satu-satunya metode penyampaian yang didukung 100% data ke program lain. (Lagi pula, jika itu disajikan sebagai file di situs web, itu paling sederhana untuk menyimpannya sebagai file lagi.)

\n\n

Demikian pula, jika Anda membuka file secara langsung dari dalam arsip, file tersebut akan dibongkar ke% TEMP% terlebih dahulu - program pengarsipan menangani dekompresi sehingga program lain tidak perlu melakukannya. Seringkali, Anda bahkan dapat mengedit file dalam program, dan itu akan dikemas ulang setelahnya. (Sebagai perbandingan, jika program punya untuk membongkar file itu sendiri, maka itu harus mengemasnya kembali setiap kali Anda menyimpan, meskipun itu sangat jarang diperlukan.)

Programmer dan power user telah menggunakan lokasi yang disebut temp atau test selama bertahun-tahun untuk menyimpan file tidak penting yang dapat dihapus setelah selesai menggunakannya.

\n\n

%temp% secara khusus berkaitan dengan Windows Temp Folder Variable, Ini adalah variabel per pengguna (pada sistem modern, defaultnya adalah %USERPROFILE%\\AppData\\Local\\Temp). Aplikasi dapat menulis ke folder ini jika mereka memiliki persyaratan file sementara.

\n\n

Contoh terbesarnya adalah installer. Banyak penginstal adalah file tunggal dan ketika Anda melihat bilah yang membutuhkan waktu lama pada penginstalan besar, biasanya ini memperluas file ke folder% temp%.

\n\n

Cukup sering saya menulis skrip batch yang membutuhkan folder sementara / pementasan. Mereka hanya kecil, dan saya biasanya selalu berakhir menggunakan md temp, gunakan itu, rd temp /s. Jika saya mendistribusikan skrip, dan itu melibatkan banyak file, akan masuk akal untuk menggunakan nama unik di folder %temp% (itu akan hanya menjadi sedikit lebih canggung untuk membuat kode). Meskipun sebagian besar orang tidak mengubah jalur default, ada banyak orang yang melakukannya, atau, lingkungan server yang Anda inginkan sesedikit mungkin pada hard drive utama.

\n\n

Adapun mengapa tidak menggunakan memori, masih berlaku hari ini - itu tidak cukup besar untuk semuanya. Laptop pertama saya memiliki memori 2 MB atau 4 MB (saya lupa), dan sementara banyak program berukuran kecil, seringkali ada beberapa yang berada dalam kisaran 5-10 MB (MS Office dulu berukuran sekitar 20-30 MB untuk seluruh suite!).

\n\n

Sekarang, rata-rata jauh lebih tinggi (mesin level awal sekitar 2-4 GB, dan pengguna daya 8-16 GB, pengguna daya dengan sedikit uang 16 GB +), dengan beberapa pengecualian seperti skrip, memori dapat benar-benar digunakan lebih banyak daripada saat ini.

\n\n

Namun, hingga mesin level awal berada dalam Rentang 16-32 GB (berikan waktu beberapa tahun!) Saya tidak berpikir kita akan memiliki banyak kemajuan.

\n\n

%temp% akan tinggal di sini untuk waktu yang lama!

Dengan perangkat lunak yang saya tulis, saya telah menggunakan folder sementara untuk memproses pesawat CVR file. File biasanya berukuran 512 MB. Selama pemrosesan file melalui berbagai format.

\n\n
    \n
  • Format A: Data kepemilikan asli.
    • \n
  • Format B: Data bertautan.
    • \n
  • Format C: Empat file G711, dan satu file CVR yang berisi data tambahan.
    • \n
  • Format D: File Four Wave, dan file CVR.
    • \n
\n\n

Saya mulai dengan menyalin file asli ke folder Temp tempat semua pemrosesan berlangsung.

\n\n

Saya menggunakan satu perpustakaan untuk mengonversi dari format A ke format C (secara otomatis melalui format B). Perpustakaan ini hanya membutuhkan nama file masukan. Ini memutuskan nama file output di folder yang sama. Itu semua di luar kendali saya.

\n\n

Saya menggunakan perpustakaan kedua untuk mengonversi dari format C ke format D.

\n\n

Direktori terakhir tempat file ditempatkan terlihat oleh pengguna, jadi saya tidak ingin mereka melihat file melalui transisi mereka, atau bagi mereka untuk melihat sebagian file, hanya hasil akhir. Setelah semua pemrosesan selesai, file yang diperlukan dipindahkan ke tujuan akhirnya dan folder Temp dihapus.

\n\n

Folder Temp menyembunyikan semua pemrosesan ini dari pengguna. Yang perlu mereka ketahui hanyalah persentase kemajuan.
