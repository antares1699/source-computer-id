Itu tergantung pada sistem file dan sakelar yang Anda gunakan dengan fsck.

\n\n

Apa fsck Periksa dan Mencoba untuk Memperbaiki memberikan penjelasan yang cukup rinci tentang apa yang dilakukan fsck ketika dipanggil tanpa sakelar.

\n\n

Beberapa pemeriksaan membutuhkan waktu yang konstan (misalnya, pemeriksaan blok super), banyak di antaranya bergantung pada jumlah inode yang harus diproses oleh fsck (misalnya, pemeriksaan jumlah tautan), yang lain pada ukuran sistem file (misalnya, data biasa cek blok).

\n\n

Tanpa sakelar -c, fsck tidak akan memeriksa konten file biasa, jadi ukuran file tidak relevan. Bahkan saat memeriksa blok yang buruk, waktu eksekusi hanya bertambah seiring dengan ukuran sistem file. Bagian mana dari partisi yang ditempati tidak mempengaruhi waktu eksekusi. 1

\n\n

Jumlah inode dari sistem file adalah konstan, yaitu diputuskan pada waktu pembuatan. Setiap metadata file disimpan di salah satu inode tersebut. Berdasarkan ext4 , inode yang tidak digunakan ditandai seperti itu di ext4, tetapi tidak di ext3. Dengan demikian, waktu eksekusi fsck meningkat dengan jumlah file di ext4, tetapi konstan (dan tinggi) di ext3.

\n\n

Gambar berikut mengilustrasikannya:

\n\n

\n\n

Perhatikan bahwa jumlah file yang berbeda diwakili oleh warna yang berbeda. Grafik untuk ext3 dengan 2,1 juta, 100 ribu dan bahkan nol file identik. Grafik untuk ext4 dengan file 2.1M dan 100k masing-masing menunjukkan perbedaan yang signifikan.

\n\n

Sebaliknya, waktu eksekusi meningkat dengan jumlah inode dengan ext3, sedangkan inode yang tidak digunakan tidak mempengaruhi waktu eksekusi dengan ext4.

\n\n
\n\n

1 Diverifikasi secara empiris dengan menjalankan fsck -c pada partisi kosong.
