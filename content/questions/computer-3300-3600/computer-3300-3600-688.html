Saya memiliki disk WD 3TB (di drive eksternal My Book Essential.) Saya menggunakan parted di Linux untuk mempartisi dengan label disk GPT dan memuatnya dengan data dari laptop saya melalui kabel USB. Kemudian saya membukanya dan beralih menjadi drive SATA internal pada komputer yang berbeda (desktop).

\n\n

Linux (Fedora 14 di kedua komputer) tidak mengenali disk. parted mengatakan label disk yang tidak dikenal. Ia juga melaporkan ukuran sektor logis sebagai 512 byte: "Ukuran sektor (logis / fisik): 512 byte / 4096 byte. Ketika saya menghubungkannya ke laptop saya melalui USB, dilaporkan sebagai 4096/4096.

\n\n

Saya mencoba mempartisi ulang dengan mengalikan semua offset sektor dengan 8, tetapi masih tidak mengenali datanya. (Bukannya aku sangat terkejut.)

\n\n

Jika saya beralih ke koneksi USB pada mesin desktop, itu beralih kembali ke sektor logis 4096 byte. Jadi tampaknya papan pengontrol USB-lah yang membuatnya menggunakan sektor logis 4096 byte. Sejujurnya, 4096 lebih masuk akal bagi saya, mengingat ini adalah ukuran sektor perangkat keras, tetapi apakah ada cara untuk membuat USB atau SATA menggunakan nilai lain?

Agak terlambat, tetapi saya memiliki masalah yang sama setelah port USB casing hard disk IOMEGA saya putus. Saya beralih ke case USB-2-SATA lain hanya untuk menemukan, bahwa saya tidak dapat memasang partisi EXT4. Untuk beberapa alasan kasus IOMEGA melaporkan ukuran sektor logis menjadi 4096, tetapi kasus baru saya mengatakan itu hanya 512 byte. Ini mengencangkan tabel partisi MS-DOS.

\n\n

Itu membuat saya gila, karena dengan bantuan testdisk saya dapat mengakses partisi, saat mengubah ukuran sektor, tetapi saya tidak menemukan cara untuk mengubah ukuran sektor di seluruh sistem. Ternyata, itu tidak perlu, karena EXT4 tidak mempedulikan ukuran sektor, Anda hanya perlu mencari awal partisi yang ingin Anda akses.

\n\n

Perbaikan cepat: Gunakan perangkat loopback untuk menggeser ke posisi partisi yang benar.

\n\n

Perbaikan permanen: Tulis ulang tabel partisi, sebagaimana mestinya.

\n\n

Dalam kasus saya, drive yang terpengaruh adalah /dev/sdb.

\n\n

Perbaikan cepat saya relatif mudah, karena saya hanya memiliki satu partisi yang dimulai di sektor 63. 

$ sudo sfdisk -d /dev/sdb\n# partition table of /dev/sdb\nunit: sectors\n\n/dev/sdb1 : start=       63, size=623708785, Id=83\n

Sekarang kita harus menghitung posisi partisi, ketika ukuran sektor masih 4096 byte: 

63 sectors * 4096 bytes = 258048 bytes\n

Dan gunakan itu dengan losetup: 

$ sudo losetup /dev/loop0 /dev/sdb -o 258048\n$ sudo mount /dev/loop0 /mnt\n

Partisi Anda sekarang harus dipasang pada /mnt.

\n\n

Untuk perbaikan jangka panjang saya menggunakan sfdisk untuk membuang tata letak partisi: 

$ sudo sfdisk -d /dev/sdb > sdb.partitions.sfdisk.text\n

Memperbaiki tabel partisi dengan mengalikan dengan sektor awal dengan 8 jika sesuai - di sini saya menyadari tabel partisi saya agak kacau dari awal.

\n\n

Saya mengedit dump tabel partisi dengan nano sdb.partitions.sfdisk.text: 

/dev/sdb1 : start=       63, size=623708785, Id=83\n

untuk: 

/dev/sdb1 : start=      504, size=623708785, Id=83\n

dan kemudian memperluas partisi untuk menggunakan semua ruang yang tersedia (yang saya tentukan dengan cara lain): 

/dev/sdb1 : start=      504, size=625141944, Id=83\n

Langkah terakhir adalah menulis kembali tabel partisi dengan: 

$ sudo sh -c 'cat sdb.partitions.sfdisk.text | sfdisk /dev/sdb'\n
