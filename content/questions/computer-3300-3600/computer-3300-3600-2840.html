Saya baru-baru ini menambahkan baris untuk stik memori USB ke /etc/fstab, agar dapat memasangnya secara otomatis, menggunakan UUID untuk mengidentifikasi perangkat.

\n\n

Sebelumnya saya sudah memiliki beberapa drive ext4 eksternal yang UUIDnya ditulis dalam huruf heksadesimal huruf kecil [0-9a-f] seperti ini: 

UUID=3eeaaa43-dead-beef-cafe-243bcaaad475 /media/external_2TB ext4 rw,user,noexec,nosuid,nodev,noatime 0 2\n

Saya memeriksa UUID stik USB saya dengan blkid, mendapatkan UUID heksadesimal huruf besar dan memasukkannya pada /etc/fstab dengan cara yang sama seperti garis ext4, dalam heksadesimal huruf kecil, seperti ini: 

UUID=125c-a3eb  /media/usb_8GB vfat defaults,users,umask=0  0 0\n\nmount: special device UUID=125c-a3eb does not exist\n

Kemudian saya mendapat ide untuk mencobanya dalam huruf besar ( gU3w di Vim) dan bekerja tanpa masalah. Untuk mengonfirmasi ini, saya melepas, dan mengubah UUID kembali ke huruf kecil, pemasangan gagal, dan kembali ke huruf besar, dan berhasil dipasang.

\n\n

Sejauh yang saya pahami, UUID adalah bilangan heksadesimal, bukan string, jadi kasus huruf seharusnya tidak relevan. Tapi jelas tidak. RFC4122 mendefinisikan UUID sebagai "case insensitive on input". Apakah ada sesuatu yang saya lewatkan atau apakah perilaku mount/umount ini bug?

\n\n

Saya menggunakan Debian GNU / Linux 7.0 Wheezy dengan versi kernel kustom 3.6.8, jika itu penting.

CATATAN: Ini adalah spekulasi, berdasarkan bagaimana perangkat biasanya diakses di Unix / Linux dan fakta bahwa nama file peka huruf besar kecil.

\n\n

Linux menggunakan UUID untuk membuat objek pada /dev/disk/by-uuid. Seperti semua "file" di Linux, nama peka huruf besar / kecil. Oleh karena itu, ketika mengacu pada disk oleh UUID, perbandingan dilakukan sebagai perbandingan nama file pada string hex daripada perbandingan biner pada data yang diwakili oleh string tersebut.
