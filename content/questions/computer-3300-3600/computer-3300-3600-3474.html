The Day ™ akhirnya tiba. Saya telah menghindari IPv6 sampai sekarang, tetapi ketidaktahuan saya yang penuh kebahagiaan harus diakhiri.

\n\n

ISP saya memberi tahu saya bahwa perangkat di jaringan saya melakukan pencarian DNS untuk salah satu server C&C yang offline dalam tindakan penegakan hukum baru-baru ini terhadap botnet Avalanche. Saya perlu menemukan perangkat itu dan menanganinya, jadi saya mengaktifkan logging di server DNS saya. Akhirnya setelah empat hari permintaan pencarian DNS yang cocok dibuat, tetapi saya kecewa permintaan itu berasal dari alamat fe80::113d:d91e:e685:943b. Sampah. Saya seorang pemula dalam hal IPv6 dan saya memiliki mesin di 60+ jaringan node saya yang merupakan bagian dari botnet yang memuntahkan malware.

\n\n

Saya menjalankan tracert dan menentukannya di tautan lokal dan saat ini sedang online: 

Tracing route to fe80::113d:d91e:e685:943b over a maximum of 30 hops\n\n  1     9 ms    <1 ms     1 ms  fe80::113d:d91e:e685:943b\n

Dengan perangkat IPv4 saya dapat melihat sewa DHCP saya untuk mendapatkan nama perangkat. Jika gagal, saya akan melakukan ping, lalu menjalankan arp -a untuk mendapatkan alamat MAC-nya, yang setidaknya memberi saya pabrikannya. Tetapi jaringan ini tidak memiliki server DHCP IPv6 dan arp sepertinya tidak menggunakan IPv6.

\n\n

Saya mencoba kursus kilat di IPv6 dan mengetahui bahwa awalan fe80 berarti alamatnya link-local dan saya bisa mendapatkan alamat MAC dari alamat tersebut. saya mencobanya dan dapatkan MAC 13:3d:d9:85:94:3b. Tidak ada alat pencarian OUI yang mengenalinya dan tidak muncul di sewa DHCP IPv4 saya.

\n\n

Bagaimana cara menentukan perangkat mana di jaringan saya yang memiliki alamat IPv6 ini?

\n\n

Server saya dan mesin tempat saya melakukan pemecahan masalah menjalankan Windows.

Terimakasih untuk komentar ini tentang Pengguna Super lainnya menjawab , Saya menemukan perintah: 

netsh int ipv6 show neighbors\n

Saya sangat senang karena mengetahui alamat MAC perangkat di tautan lokal, seperti yang dilakukan arp -a: 

Interface 10: Local Area Connection\n\nInternet Address                              Physical Address   Type\n--------------------------------------------  -----------------  -----------\n<redacted>\nfe80::113d:d91e:e685:943b                     4c-72-b9-d2-b5-5d  Reachable\n<redacted>\n

Saya kemudian dapat membandingkan Alamat Fisik dengan sewa IPv4 di server DHCP saya dan mendapatkan nama NetBIOS dari perangkat yang melanggar. Mudah sekali.
