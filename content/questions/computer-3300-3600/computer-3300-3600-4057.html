Bagaimana saya bisa mengakses mini-pc saya melalui ethernet / wifi tanpa memiliki Monitor?

\n
\n

Tergantung bagaimana Anda mendefinisikan "akses".

\n

Jika Anda tidak memiliki kernel & root filesystem (RFS) yang terbukti untuk boot, maka Anda benar-benar perlu memiliki menghibur terpasang ke miniPC ini. Jika tidak, Anda tidak akan dapat melihat semua pesan startup yang dibuat. (Setelah kernel diuji, kernel disiapkan untuk boot diam mode untuk rilis. Meskipun kernel Windows lebih bertele-tele dari biasanya saat menggunakan "mode aman".)

\n

PC biasanya menggunakan perangkat video (misalnya monitor monokrom / CGA / EGA / xVGA) sebagai konsol (teks dan / atau GUI). PC tanpa kepala dan sistem tertanam biasanya menggunakan port serial (berbiaya rendah) (UART / USART) sebagai konsol (hanya teks). Linux dapat menggunakan konsol melalui adaptor USB-ke-serial, tetapi beberapa pesan startup akan hilang karena inisialisasi yang terlambat dari tumpukan USB.

\n

Login jarak jauh seperti telnet atau ssh hanya bisa terjadi setelah kernel berhasil di-boot . Jika Anda tidak memiliki konsol dan kernel gagal untuk boot, maka Anda akan memiliki sedikit informasi untuk menjelaskan atau men-debug kegagalan boot.

\n

Dengan kata lain, bayangkan Anda mendapatkan laptop baru dengan SSD yang layarnya tidak menyala hingga Win7 atau Linux meminta nama pengguna / kata sandi Anda. Kode bip BIOS POST dinonaktifkan. Tidak ada LED aktivitas "disk". Saat pertama kali menyalakan laptop, tidak ada yang muncul di layar sama sekali. Tidak ada yang menunjukkan kemajuan boot. Bagaimana Anda menjelaskan kegagalan boot? Bagaimana Anda mulai mengatasi ini? Berapa banyak waktu yang Anda miliki untuk mencoba berbagai kemungkinan perbaikan? Atau apakah Anda hanya akan menggunakannya sebagai palang pintu?

\n

BTW jika LAN Anda (relatif) aman, maka yang Anda perlukan untuk mengakses miniPC (berjalan) adalah telnet (daripada ssh ). Mulai telnetd.dll daemon (atau konfigurasi inetd ) di miniPC, dan gunakan a telnet klien di PC host Anda.

\n

Booting dari LAN tidak menyelesaikan atau mengatasi masalah konsol ini sama sekali. LANboot berarti bahwa image kernel diperoleh dari server, bukan penyimpanan lokal.\nIni tidak masuk akal bila ada gigabyte lokal (dan dapat dilepas ) kapasitas penyimpanan. Menguji kernel dengan LANbooting memang nyaman. U-Boot mampu menggunakan LAN dan tftp , serta penyimpanan lokal untuk memuat imej kernel; "U" dalam namanya berarti "universal". Perhatikan bahwa LAN juga dapat digunakan untuk mengakses sistem file root melalui NFS .

\n

Memperbarui

\n

Jika komputer papan tunggal Anda memiliki UART atau USART, maka konsol port serial adalah cara yang tepat. Namun komplikasi yang mungkin terjadi adalah tidak ada konektor port serial DB-9 standar atau, yang lebih umum, hanya sinyal level logika yang terpapar daripada sinyal RS-232. Beberapa perangkat keras tambahan kemudian diperlukan untuk menggunakan antarmuka ini sebagai konsol.

\n

Untuk Raspberry Pi dan Mele A100 / 1000/2000, ada UART tetapi hanya sinyal level logika yang tersedia. Papan atau konverter eksternal harus dihubungkan, yang akan menyediakan koneksi DB-9 atau USB ke PC host (yang akan membutuhkan program emulator terminal, seperti Istilah Tera atau Dempul ). Tutorial untuk mengatur port serial pada Raspberry Pi adalah sini . U-Boot mungkin akan membutuhkan recomplie untuk menggunakan konsol port serial. Dukungan untuk konsol serial juga akan dikompilasi ke dalam kernel; itu konsol serial untuk Linux ditentukan di "baris perintah kernel" yang berasal dari U-Boot. Linux memungkinkan spesifikasi beberapa konsol, dan pesan keluaran akan ditampilkan pada semuanya.

\n

Tambahan

\n

Jika konsol port serial tidak tersedia dan / atau akses internet adalah suatu keharusan, maka ada solusi perantara untuk mem-boot kernel Linux dengan netconsole . Tapi itu biasanya membutuhkan LAN yang aman dan alamat IP yang diketahui (misalnya statis) yang ditetapkan ke host yang bertindak sebagai "konsol".
Tetapi untuk visibilitas penuh dari proses boot, Anda ingin menggunakan konsol serial, karena hampir semua SBC berkemampuan Linux memiliki UART (dengan level logika atau EIA / RS-232) untuk konsol port serial.
